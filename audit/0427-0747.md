## 실행 요약

코드 기준 판정은 **“프로덕션 운영 가능 기반은 있으나, 고가용·고신뢰 프로덕션에는 조건부 미달”**입니다. 이유는 단순 기능 부족이 아니라, 여러 핵심 전이가 **파일시스템·D1/SQLite·Postgres·Redis·R2·Chroma·GitHub·LLM·Piston**에 걸쳐 있으면서도 다중 저장소 트랜잭션, 재시도 멱등성, 동시성 제어, 장애 격리가 일관되게 설계되어 있지 않기 때문입니다.

읽은 범위는 Express backend, Cloudflare Worker API Gateway, R2/SEO/Terminal gateway, terminal-server, shared route boundary, config, repositories, services, migrations, 테스트입니다. 전체 엔드포인트는 실제 라우트 파일과 조건식 fetch handler까지 보정해 **434개 operation**으로 정리했습니다. Worker-native route와 backend-origin route가 같은 public path를 공유하는 경우도 **서로 다른 handler**로 별도 행에 넣었습니다.

전체 엔드포인트 표는 여기 있습니다. Method, Path, Handler, Service, 입력/출력, 인증/인가, 상태변경, 부수효과, 실패 조건, 멱등성, 트랜잭션, 근거 파일/라인을 포함했습니다.

## 시스템 아키텍처 개요

실제 구조는 **Edge Worker + Origin Backend + 다중 저장소/외부 API 혼합형 아키텍처**입니다.

| ---------------------- | ----------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------- |

| Cloudflare API Gateway | public API 진입점, CORS, IAP JWT 검증, route boundary, 일부 native D1/KV/R2 처리, 일부 backend proxy                         | `workers/api-gateway/src/index.ts:182-276`, `workers/api-gateway/src/routes/registry.ts:44-127`                           |

`workers/api-gateway/src/index.ts`는 모든 요청에 CORS, IAP JWT, logger를 적용하고 `/api/v1/ai/*`, `/api/v1/chat/*`에 tracing middleware를 붙입니다. 그 뒤 Worker route registry를 `/api/v1`에 mount하고, 매칭되지 않는 요청은 `proxyToBackend()`로 origin에 전달합니다. 이 proxy는 `BACKEND_ORIGIN`이 없으면 500, route ownership이 허용되지 않으면 404, backend 요청 시 `X-Backend-Key`를 삽입합니다. 근거는 `workers/api-gateway/src/index.ts:37-160`, `:182-276`입니다.

Backend origin은 public health/config와 public auth/notification router를 먼저 mount한 뒤, `/metrics`에는 backend key를 요구하고, 이후 전역 `requireBackendKey`를 적용합니다. protected registry는 그 뒤에 mount됩니다. 근거는 `backend/src/index.js:85-130`입니다. 따라서 backend route 중 `/api/v1/auth`와 `/api/v1/notifications`는 public registry에 속하고, 나머지는 전역 backend key 뒤에 있습니다. `notifications`의 `/stream`, `/unread`, `/history`, `/:id/read`는 backend-key가 아니라 route-level `requireUserAuth`가 직접 적용됩니다. `/push`, `/outbox/internal`, `/health`는 route 내부에서 `requireBackendKey`를 다시 요구합니다. 근거는 `backend/src/routes/registry.js:28-31`, `backend/src/routes/notifications.js:127-305`입니다.

### 1. Worker-native `POST /api/v1/ai/auto-chat`

**Before**

```json
{
  "headers": {
    "Authorization": "Bearer <edge JWT>"
  },
  "body": {
    "messages": [
      { "role": "user", "content": "..." }
    ],
    "temperature": 0.7,
    "maxTokens": 1000
  }
}
```

**흐름**

1. Worker app-level middleware가 CORS, IAP JWT, logger, AI tracing을 적용합니다. 근거: `workers/api-gateway/src/index.ts:182-186`.
2. `/api/v1` registry에서 `ai` router가 native로 처리합니다. 근거: `workers/api-gateway/src/index.ts:271-276`, `workers/api-gateway/src/routes/registry.ts:44-48`.
3. `ai.post('/auto-chat', requireAuth, ...)`가 edge JWT를 요구합니다. 근거: `workers/api-gateway/src/routes/ai.ts:303-304`.
4. `rejectOversizeJson()`이 `content-length` 기준 128KB 초과를 거부합니다. 근거: `workers/api-gateway/src/routes/ai.ts:13-36`.
5. `enforceAiRateLimit()`가 KV key `ratelimit:ai:auto-chat:<subject>`를 읽고 60초 TTL로 count를 증가시킵니다. 근거: `workers/api-gateway/src/routes/ai.ts:40-55`.
6. `validateChatMessages()`가 messages 배열을 검사합니다. 근거: `workers/api-gateway/src/routes/ai.ts:68-70`, `:309-315`.
7. `createAIService(c.env).chat()`가 LLM provider를 호출합니다. 근거: `workers/api-gateway/src/routes/ai.ts:317-323`.

**After**

```json
{
  "ok": true,
  "data": {
    "content": "<assistant text>",
    "model": "<provider model>",
    "provider": "<provider>"
  }
}
```

**상태 변화**

* KV rate-limit counter가 증가합니다.
* LLM provider 호출이 발생합니다.
* DB transaction은 없습니다.

**멱등성**

* 미보장입니다. 같은 요청 재시도 시 KV counter가 추가 증가하고 LLM 호출도 반복됩니다.

**실패 조건**

* 인증 실패, body oversize, rate limit, invalid messages, provider failure.

### 2. Backend-origin `POST /api/v1/posts`

**Before**

```json
{
  "headers": {
    "X-Backend-Key": "<worker inserted>",
    "Authorization": "Bearer <admin JWT or admin bearer>"

  "body": {
    "title": "Post title",

    "content": "markdown body",

      "tags": [],
      "category": "General",

    }
  }
}
```

**흐름**

1. Worker fallback proxy는 route ownership 허용 시 origin으로 전달하고 `X-Backend-Key`를 삽입합니다. 근거: `workers/api-gateway/src/index.ts:55-79`.
2. Backend는 public route 이후 전역 `requireBackendKey`를 적용합니다. 근거: `backend/src/index.js:113-130`.
3. `POST /api/v1/posts`는 `requireAdmin`을 요구합니다. 근거: `backend/src/routes/posts.js:327`.
4. year는 `YYYY`, slug는 `slugify()` 후 filename regex로 검증됩니다. 근거: `backend/src/routes/posts.js:329-338`, `:74-79`.
5. `frontend/public/posts/<year>/<slug>.md`가 존재하면 409입니다. 근거: `backend/src/routes/posts.js:340-345`.
6. frontmatter와 body를 markdown으로 조립해 파일을 씁니다. 근거: `backend/src/routes/posts.js:346-357`.
7. per-year manifest와 unified manifest를 재생성하고 posts cache prefix를 invalidate합니다. 근거: `backend/src/routes/posts.js:359-364`.

**After**

```json
{
  "ok": true,
  "data": {
    "path": "/posts/2026/post-title.md",
    "manifestTotal": 123
  }
}
```

**상태 변화**

* 없음 → markdown file 생성.
* `manifest.json`, `posts-manifest.json`, `posts/posts-manifest.json` 갱신.
* posts cache 무효화.

**불변식**

* year는 4자리 숫자.
* filename은 `/^[a-zA-Z0-9][a-zA-Z0-9\-_]*\.md$/`.
* 같은 year/slug 파일은 중복 생성 불가.

**트랜잭션/동시성**

* 파일 write, manifest write, cache invalidation이 하나의 트랜잭션으로 묶여 있지 않습니다.
* `exists(abs)` 후 `writeFile(abs)` 사이 race가 있습니다. 동시 생성 요청은 둘 다 exists=false를 볼 수 있습니다.

## 아키텍처 책임 분리 평가

### 적절한 부분

* **Route boundary contract**가 Worker와 backend 모두에 있습니다. Worker는 `buildRouteBoundaryHeaders`, backend는 `attachBoundaryHeaders()`를 사용합니다. 근거: `workers/api-gateway/src/routes/registry.ts:87-127`, `backend/src/routes/registry.js:74-98`.
* backend-key, admin auth, user JWT auth가 분리되어 있습니다. 근거: `backend/src/middleware/backendAuth.js`, `adminAuth.js`, `userAuth.js`.
* analytics `recordVisit()`는 Postgres transaction을 사용합니다. 근거: `backend/src/repositories/analytics.repository.js:52-70`.
* AI client에는 circuit breaker와 health cache가 있습니다. 근거: `backend/src/services/ai/openai-client.service.js:94-159`.
* SSRF 방어가 vision URL 처리에 있습니다. private IP/blocked hostname 검사를 수행합니다. 근거: `backend/src/routes/ai.js:503-590`.

### 책임 누수 / 결합도 / 경계 붕괴

1. **Route handler가 use-case/service 역할을 과도하게 수행합니다.**
   posts, memos, memories, user, debate, images는 route 파일 안에서 validation, state transition, repository write, external side effect를 직접 수행합니다. 예: posts create는 `routes/posts.js:327-365`에서 파일 write, manifest regenerate, cache invalidate까지 수행합니다.

2. **Worker-native와 backend-origin endpoint가 중복됩니다.**
   `/api/v1/ai`, `/api/v1/user`, `/api/v1/memos`, `/api/v1/memories`, `/api/v1/rag` 등은 Worker에도 있고 backend에도 있습니다. public effective route는 Worker registry가 먼저 잡지만, origin이 직접 노출되면 다른 auth/state path가 열립니다. backend nginx가 80/443/8443을 publish합니다. 근거: `backend/docker-compose.yml:85-88`.

3. **동적 AI config가 backend → Worker internal API에 의존합니다.**
   backend AI service가 Worker `/api/v1/internal/ai-config`를 source priority 1로 둡니다. 근거: `backend/src/services/ai/dynamic-config.service.js:1-10`. 이는 origin service가 edge config에 런타임 의존하는 숨겨진 역방향 결합입니다.

4. **state가 process-local인 서비스가 많습니다.**
   auth challenge/OAuth state fallback, chat sessions, debate sessions, live room history, prompt overrides 등이 process memory입니다. multi-instance, restart, rolling deploy에서 상태 일관성이 깨집니다.

5. **외부 API 전이가 outbox/compensation 없이 inline으로 수행됩니다.**
   GitHub PR, R2 upload, Chroma upsert, OpenNotebook, Vercel hook, Piston execute가 대부분 요청 thread에서 직접 수행됩니다.

## 운영/관리 방안 평가

### 설정/시크릿 관리

Backend는 zod config schema로 env를 검증합니다. `BACKEND_KEY`, `JWT_SECRET`, admin secret 누락은 protected env에서 error로 처리됩니다. 근거: `backend/src/config/schema.js:4-94`, `backend/src/config/index.js:217-260`.

Worker는 wrangler secrets를 문서화하고 D1/R2/KV binding을 production/staging에 분리합니다. 근거: `workers/api-gateway/wrangler.toml:13-50`, `:84-124`.

위험은 두 가지입니다.

* backend `config` admin route가 `.env` 저장을 수행합니다. 운영에서 local env mutation은 rollback과 감사 추적이 약합니다. 근거: `backend/src/routes/config.js:460-521`.
* TOTP setup도 `.env`에 `TOTP_SECRET`을 씁니다. 실패 시 memory rollback은 있지만, secret lifecycle/rotation 관점에서는 운영 secret store와 분리되어 있습니다. 근거: `backend/src/routes/auth.js:303-315`.

### 로깅/메트릭/트레이싱

* backend는 morgan과 Prometheus request duration/count metric을 둡니다. 근거: `backend/src/index.js:72-82`.
* `/metrics`는 backend-key 보호됩니다. 근거: `backend/src/index.js:115`.
* Worker는 logger middleware와 AI/chat tracing middleware를 둡니다. 근거: `workers/api-gateway/src/index.ts:182-186`.
* Cloudflare Worker observability가 enabled입니다. 근거: `workers/api-gateway/wrangler.toml:108-110`.

공백은 distributed tracing입니다. `X-Request-ID`는 Worker proxy에서 생성하지만, backend service/external calls까지 일관된 trace span으로 이어지는 구조는 보이지 않습니다. 근거: `workers/api-gateway/src/index.ts:81-85`.

### 장애 대응성

강점:

* Worker proxy는 backend 502~504를 JSON + `Retry-After: 30`으로 변환합니다. 근거: `workers/api-gateway/src/index.ts:135-152`, `backend-proxy.ts:193-211`.
* backend readiness는 PG migration failure를 degraded로 표시할 수 있습니다. 근거: `backend/src/index.js:39-49`, `:97-103`.
* AI OpenAI-compatible client는 circuit breaker를 갖습니다. 근거: `openai-client.service.js:94-159`.

약점:

* live Redis bridge는 한번 실패하면 `liveRedisBridgeFailed=true` 후 재시도하지 않습니다. 근거: `backend/src/services/live-chat.service.js:683-717`.
* D1/SQLite app-level transaction API가 없습니다. migrations에는 transaction이 있지만 application write flow는 개별 statement 중심입니다. 근거: `backend/src/repositories/base/d1.repository.js:61-67`, `:87-123`.
* GitHub/R2/Chroma/LLM/Piston 장애에 대한 outbox + retry + compensation이 일관되지 않습니다.

### 배포/롤백

Docker compose는 api, nginx, terminal-server, workers-local, chroma, redis, postgres healthcheck를 정의합니다. 근거: `backend/docker-compose.yml:37-200`. Worker production/staging binding도 있습니다. 근거: `workers/api-gateway/wrangler.toml:84-124`.

하지만 schema migration rollback, posts manifest rollback, GitHub PR partial failure rollback, Chroma/R2 compensation이 없습니다. 배포 자체는 가능하지만 상태 전이 rollback 설계는 약합니다.

### 캐시/큐/외부 API 장애 격리

* HTTP cache middleware와 Redis queue/DLQ가 존재합니다.
* AI async mode는 Redis availability를 체크합니다. 근거: `backend/src/services/ai/ai.service.js:94-132`.
* Worker cron은 artifact/notification outbox flush를 수행합니다. 근거: `workers/api-gateway/src/index.ts:345-355`.

하지만 idempotency key 없는 POST가 많고, external side effect가 request inline으로 묶여 있어 장애 격리는 제한적입니다.

## 구체적 개선안

1. **Origin 접근 경계를 먼저 고정해야 합니다.**
   backend origin은 Worker 외부에서 접근 불가하도록 firewall/mTLS/IP allowlist를 적용하고, backend public registry도 edge-signed header 또는 backend-key를 요구하도록 통일하십시오. 특히 `/api/v1/chat/ws` upgrade에는 JWT 또는 backend-key 검증을 추가해야 합니다.

2. **idempotency key를 표준화하십시오.**
   `POST /analytics/view`, `POST /posts`, `POST /images/upload`, `POST /notifications/outbox/internal`, AI generation, debate vote, user session create에 `Idempotency-Key`를 받고 D1/KV/Postgres에 결과 cache/dedupe를 저장하십시오.

3. **D1/SQLite application transaction wrapper를 추가하십시오.**
   현재 D1 repository는 migration transaction만 있고 application write는 개별 statement입니다. `transaction(fn)`을 추가하고 user session create/recover, memo version update, memory batch, notification outbox/inbox materialize에 적용해야 합니다.

4. **cross-store 전이는 outbox로 바꾸십시오.**
   GitHub PR, R2 upload 후 vision, Chroma upsert, Vercel deploy hook, notification broadcast는 inline completion 대신 durable outbox + worker/cron retry + 상태 machine을 사용해야 합니다.

5. **process-local state를 durable state로 옮기거나 명시적으로 single-instance로 제한하십시오.**
   chat sessions, debate sessions, live rooms, auth challenge/OAuth state fallback, prompt overrides는 Redis/D1에 저장하거나 “single instance only” 운영 제약을 명문화해야 합니다.

6. **live Redis bridge retry를 고치십시오.**
   `liveRedisBridgeFailed`가 영구 latch가 되지 않게 exponential backoff retry를 넣고, readiness에 Redis pubsub 상태를 노출하십시오.

7. **memo version semantics를 수정하십시오.**
   `PUT memo`에서 version snapshot은 next content가 아니라 previous content 또는 명확한 revision model이어야 합니다. `memo_versions` insert와 `memo_content` update를 같은 transaction으로 묶으십시오.

8. **debate round transition bug를 수정하십시오.**
   현재 round endpoint는 `currentRound`를 증가시킨 뒤 limit 초과를 검사하는 구조입니다. 초과 여부를 먼저 검사하고 상태를 변경해야 합니다. vote에는 voter/session identity dedupe를 추가하십시오.

9. **AI model override를 origin에서 신뢰하지 마십시오.**
   `X-AI-Model`은 Worker가 서명한 header이거나 internal-only header일 때만 수용하십시오. 일반 client header는 backend에서 제거하거나 거부해야 합니다.

10. **테스트를 state transition 중심으로 재구성하십시오.**
    각 service별로 happy path보다 rollback, concurrency, idempotency, external failure를 우선 추가해야 합니다. 특히 Worker→Backend E2E stream/proxy test와 notification/chat/posts/user-session 통합 테스트가 우선순위입니다.
