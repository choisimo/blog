1편에서 우리는 파이썬 코드가 바이트코드로 번역되고 Frame 위에서 어떻게 도는지 논리적인 실행 흐름을 간략하게 보았습니다. 
그리고, 모든 동적 상태를 실체화하기 위해서 물리적인 메모리의 극한 상황에서의 예상되는 문제점 대해 질문 해보았습니다.

1편의 질문을 재정의 해보자면, "만약 프레임을 만들 메모리조차 고갈된다면 어떻게 될까?" 라고 할 수 있습니다.
낙타 혹 위의 마지막 깃털처럼 (무너지기 직전) 바로 메모리 에러 객체를 던지면서 종료되어 버릴까요?

### 1. Python 의 모든 것은 PyObject (객체) 이다.
파이썬 공식 언어 문서에는 "모든 데이터, 심지어 코드까지도 객체로 표현한다." 라고 적혀있을만큼 파이썬은 객체에 진심인 언어로 보입니다. CPython 으로 내려가서 볼 때에, 파이썬의 모든 변수와 값은 결국 **PyObject 를 가리키는 포인터** 입니다.

![](/images/2026/python-2/2.png)

메모리에 올라간 모든 파이썬 객체들은 그 앞부분에 공통 Header 를 갖습니다. 이 헤더의 핵심 필드는 Reference Count (ob_refcnt), Type Pointer (ob_type) 이 두가지 입니다. (추가적으로, 여기에 리스트나 문자열처럼 길이가 가변적인 객체는 PyVarObject 라는 구조체로 확장되어 길이를 나타내는 ob_size 필드를 추가로 가집니다.)

##### 그럼 a + b 연산은 실제로 어떻게 일어나는 것이지?
객체의 실제 동작 (+, [], 호출, 해시, 속성 접근 등)은 PyTypeObject 안에 미리 정의된 다수의 C 함수 포인터 배열인 슬롯 (tp slots) 으로 구현됩니다.

![](/images/2026/python-2/8.png)
즉, a + b 와 같은 단순한 실행 중 - 내부에서는 객체 포인터를 따라가서 -> 타입 포인터를 확인하고 -> 그 타입에 정의된 더하기 슬롯으로 디스패치한다. - 의 무거운 과정을 거칩니다.

> 공짜 점심은 없듯이, 편리한 동적 타이핑의 유연함도 공짜는 아니라는 것을 알 수 있습니다.

### 2. 네임스페이스와 속성 조회

![](/images/2026/python-2/7.png)

모든 것이 객체라면, 그 객체들의 이름은 어디에 저장되는 것인가?
네임스페이스는 PyObject 모델 위에 놓여진 거대한 딕셔너리입니다.

모듈은 네임스페이스를 딕셔너리로 가지고 있어서, m.x 를 호출하는 것은 사실상 m.__dict__["x"] 를 조회하는 것과 같습니다. 클래스나 인스턴스 역시도 __dict__ 에 자신의 속성을 저장합니다. (흔하게 사용하는 방식)

![](/images/2026/python-2/3.png)
저장을 했다면, 꺼내서 사용해야 합니다. 그럼 속성 조회를 어떻게 해야하는 걸까요? 
파이썬의 공식 descriptor 규칙에 따라 저장과는 다르게 훨씬 복잡한 로직을 걸쳐 조회가 이루어집니다. 

```text
Data descriptor → Instance dict (__dict__) → Non-data descriptor → Class variable → __getattr__()
```
만약 수백, 수천만 개의 작은 객체를 만들 때 모든 인스턴스가 각자의 __dict__ 를 가지게 된다면 메모리 오버헤드가 서버 관리자 머리에서 열로 나오는 진풍경을 볼 수 있을겁니다. 분명히 다른 효율적인 방안이 필요합니다. 

이러한 구조적 비효율성을 극복하기 위해 파이썬이 제공하는 대표적인 아키텍처 레벨의 해결책이 두 가지 있습니다.

##### 1) __slots__를 통한 딕셔너리 억제

클래스에 __slots__를 선언하면, CPython은 해당 인스턴스의 __dict__ 생성을 아예 막아버립니다. 대신 미리 정의된 이름표 개수만큼만 메모리에 고정 슬롯(배열)을 잡습니다. 객체의 크기가 극적으로 줄어들 뿐만 아니라, 딕셔너리 해싱을 거치지 않고 메모리 오프셋으로 바로 접근하므로 조회 속도도 훨씬 빨라집니다.

#### 2) 문자열 인터닝 (String Interning)

파이썬은 네임스페이스가 거대한 딕셔너리이므로, 필연적으로 문자열 키(Key)를 비교하는 연산이 쉴 새 없이 일어납니다. 이를 최적화하기 위해 자주 쓰이는 짧은 문자열이나 변수 이름들은 메모리에 딱 하나만 만들어두고 '공유'합니다.
덕분에 딕셔너리에서 키를 찾을 때 무거운 문자열 값 비교(a == b) 대신, 단순히 메모리 주소(포인터)가 같은지(a is b)만 비교하여 속도를 비약적으로 끌어올립니다.


### 3. 모든 것이 동적 객체라면, 메모리 할당의 지옥도가 펼쳐질까?
모든 것이 동적 객체라면, 이걸 담을 메모리(Heap)는 어떻게 가져올까요? C 언어처럼 매번 정수 1을 만들 때마다 OS에 malloc()을 호출해 달라고 하면(Syscall), 컨텍스트 스위칭 비용 때문에 프로그램이 지독하게 느려질 것입니다. 

그래서 CPython은 객체 크기에 따라 메모리 할당 전략을 이원화합니다.

    큰 객체 (512 bytes 초과): OS의 메모리 할당기(malloc, mmap)를 직접 사용합니다.

    작은 객체 (512 bytes 이하): 파이썬 코드의 90% 이상을 차지하는 작은 객체들을 위해 **pymalloc**이라는 자체 메모리 할당기를 씁니다.
    
    - 64비트 시스템 기준으로 CPython은 OS로부터 1MiB짜리 거대한 메모리 운동장(Arena)을 통째로 미리 빌려옵니다. 그리고 이 운동장을 4KB 단위의 풀(Pool)로 쪼갠 뒤, 고정된 크기의 블록(Block) 단위로 파이썬 객체들에게 매우 빠르게 나누어줍니다. OS와 소통하는 횟수를 최소화하여 시스템 아키텍처의 병목을 우회하는 것입니다.
  
### 4. 생명주기 관리: 객체의 죽음, 그 너머 
메모리를 빌려왔다면 다 쓴 다음에 **빌려온 것**만 다시 반납해야 합니다. 
따라서 메모리 해제는 생성보다 더 까다로운 검증이 필요할 것이 분명합니다.

![](/images/2026/python-2/4.png)
#### 1차 방어선 : Reference Counting (참조 횟수)

모든 PyObject 헤더에 들어있던 ob_refcnt가 여기서 쓰입니다.
객체를 변수에 할당하거나 리스트에 넣으면 카운트가 +1 되고, 스코프를 벗어나거나 del을 호출하면 -1이 됩니다. 이 숫자가 정확히 0이 되는 즉시, 해당 타입의 할당 해제(Deallocator) 함수가 호출되어 메모리에서 즉각 소멸합니다. 실시간으로 깔끔하게 정리되는 가장 이상적인 방법입니다.

#### 후처리 작업 : 가비지 컬렉터 (Cyclic GC)

하지만 참조 카운트만으로는 치명적인 약점이 있습니다. 바로 A 객체가 B를 바라보고, B 객체가 다시 A를 바라보는 순환 참조(Reference Cycle) 상황입니다. 외부에서 이 둘로 접근하는 변수를 지워버려도 서로를 붙잡고 있어서 카운트가 영원히 0이 되지 않습니다.

이런 "메모리 누수"를 막기 위해 파이썬은 주기적으로 순환 참조 가비지 컬렉터(GC)를 돌려 고립된 객체 섬을 찾아내 강제로 철거합니다.

### 5. 참조 카운트와 순환 참조
#### 5-1. Reference Count의 증감 확인

`sys.getrefcount()`를 사용하면 현재 객체가 몇 번 참조되고 있는지 훔쳐볼 수 있습니다.

```python
import sys

data = [1, 2, 3]
print(sys.getrefcount(data))  # 일반적으로 2 (self + refarg)
```

실행 결과:
```text
2
```

#### 5-2. 순환 참조는 Refcount만으로 치울 수 없다

GC가 나서야만 하는 극한의 순환 참조 상황을 연출해 보겠습니다. `weakref`를 사용하면 참조 카운트를 올리지 않고 객체의 생존 여부만 추적할 수 있습니다.

```python
import gc
import weakref

class Node:
    def __init__(self, val):
        self.val = val
        self.ref = None

a = Node("A")
b = Node("B")
a.ref = b
b.ref = a

a_weak = weakref.ref(a)
print(a_weak() is not None)  # True

del a, b
print(a_weak() is not None)  # True (GC 전까지)

gc.collect()
print(a_weak() is not None)  # False (GC 후)
```

실행 결과:
```text
True
True
False
```

분명히 밖에서 `del a, b`를 호출했는데도, GC가 실행되기 전에는 두 객체가 끈질기게 메모리에 살아있습니다(True). 수동으로 `gc.collect()`를 호출하여 순환 고리를 강제로 끊어주고 나서야, 비로소 객체가 깔끔하게 `None`으로 사라집니다.

---

지금까지 파이썬의 근간인 PyObject 구조부터 시작해 딕셔너리 기반의 네임스페이스(__dict__), 전용 메모리 할당기(pymalloc), 그리고 객체의 생명주기를 관리하는 참조 카운트와 가비지 컬렉터(GC)까지 살펴보았습니다.

결국, 파이썬의 핵심인 "모든 것을 동적 객체로 다루는" 개념은 개발자에게 엄청난 생산성과 유연성을 주지만, 동시에 C 언어 레벨로 내려가면서 끊임없는 포인터 추적, 메모리 할당 및 해제 오버헤드, 그리고 떨어지는 캐시 지역성(Cache Locality)이라는 막대한 비용을 치르고 있다는 겁니다.


이런 무거움을 줄이기 위해 CPython 에서는 pymalloc, 문자열 인터닝, 적응형 바이트코드 같은 꼼수들을 사용하였습니다. 하지만, 꼼수들을 어떻게 사용하여야 이 무거운 동적 객체들을 피할 수 있을까요?
