## 1. 운영 모델의 뼈대: 이데아와 현실의 끝없는 투쟁



![](/images/2026/k8s-start/1.png)


### Object / Spec / Status

쿠버네티스 객체(Object)는 클러스터에 대해 “반드시 이렇게 존재해야 한다”고 선언하는 절대적인 이데아(Idea)의 기록이다. spec은 우리가 열망하는 완벽한 이상향(Desired State)이고, status는 현재 직면한 척박한 물리적 현실(Actual State)이다.

이러한 분리가 중요한 이유는, 장애가 발생했을 때 엔지니어의 손으로 Pod 하나를 멱살 잡아 살려내는 일회성 조치가 무의미해지기 때문이다. 본질은 어떤 상위 객체가 이 시스템에 어떤 상태를 끊임없이 강제하고 있는가를 꿰뚫어 보는 데 있기에 무너진 모래성을 임시방편으로 보수해 봐야, 시스템의 통제권을 쥔 컨트롤러는 기어코 원래의 설계도(spec)대로 파도를 일으키며 모래 위에 세워진 현실을 덮어써 버린다.



### Controller: 보이지 않는 손

컨트롤러는 멈추지 않는 제어 루프(Control Loop)다. 이름이 암시하듯, 이들은 클러스터라는 세계를 끊임없이 감시하며 현실(status)이 이상(spec)에서 벗어나는 순간 가차 없이 개입하여 상태를 교정하는 전능한 관제탑이다.

“왜 지워도 리소스가 자꾸 부활하지?” “왜 내가 수정한 설정이 원상복구 되지?”
운영 중 마주하는 이 기괴한 현상들의 배후에는 언제나 컨트롤러가 있다. 인간이 터미널 앞에서 명령을 내리며 시스템을 지배하는 것 같지만, 실제 이 세계의 최종 결정권은 객체의 운명을 손에 쥔 컨트롤러에게 위임되어 있는 것이다. 


### Label / Selector / Annotation: 새겨진 운명

![](/images/2026/k8s-start/2.png)

Label은 객체의 식별과 운명을 결정짓는 키-값 메타데이터이고, Selector는 그 Label을 쥐고 시스템 안에서 자신의 통제 대상을 족집게처럼 찾아내는 그물망이다. Service, ReplicaSet, NetworkPolicy 같은 클러스터의 핵심 신경망들은 모두 이 Selector의 논리적 결합에 완벽히 의존한다.

반면 Annotation은 식별력이 거세된 비식별성 메타데이터다. 이는 시스템의 통제(Selector)를 받기 위함이 아니라, 인간이나 외부 도구를 위해 남겨두는 부가적인 주석에 불과하다. 이 둘의 경계를 허물고 섞어 쓰는 순간, "인간이 읽을 설명서"와 "시스템이 작동할 기어"가 맞물려 치명적인 장애를 낳는다.

가장 직관적인 비유를 들자면 이러하다.

    마감 세일 시간에 매대에 덩그러니 놓인 개별 상품들은 Pod (객체)

    상품 겉면에 붙은 '마감 세일 30%' 공식 바코드 스티커는 Label (라벨)

    직원의 수거 지시서("바코드가 30%인 제품만 창고로 옮겨라")는 Selector (셀렉터)

    직원이 다음 교대자를 위해 붙여둔 포스트잇("이 제품은 사은품 동봉 요망")은 
    Annotation (어노테이션)이다.


### Finalizer: "삭제"라는 환상과 etcd의 현실

쿠버네티스에서 kubectl delete를 입력하고 deleted라는 응답을 받았다고 해서, 리소스가 물리적으로 즉시 소멸했다는 뜻은 아니다. 이는 쿠버네티스의 선언적 비동기 모델이 만들어내는 정교한 환상이다.

Finalizer는 리소스의 metadata.finalizers 배열에 새겨지는 죽음의 족쇄다. 삭제 요청이 들어오면 API Server는 etcd에서 레코드를 지우는 대신, 무심하게 metadata.deletionTimestamp(사망 선고 시간)만 찍어둔다.

컨트롤러들은 이 낙인이 찍힌 객체를 발견하면 상태를 Terminating으로 바꾸고, 외부 로드밸런서 반납이나 스토리지 해제 같은 현세의 빚(정리 작업)을 청산하기 시작한다. 모든 빚을 청산하고 Finalizer 배열에서 자신의 이름을 스스로 지워냈을 때, 이 배열이 완전히 텅 비어야만 비로소 API Server는 etcd에서 이 존재를 영구히 소멸시킨다. 만약 스토리지 장애 등으로 빚을 갚지 못하면, 그 객체는 영원히 구천을 떠도는 Terminating 상태로 남게 된다.


## 2. 워크로드 컨트롤러: 계층적 상태 관리

워크로드 컨트롤러를 지배하는 것은 단 하나이다. 최상위 객체가 직접 흙을 빚어 생명을 창조하지 않고, 하위 객체에게 권한을 내려보내는 '계층적 위임(Hierarchical Delegation)'의 구조를 가지는 것이다.

### Deployment와 ReplicaSet: 상태가 없는 존재의 참을 수 없는 가벼움


Deployment는 덧없는 Pod의 생사에 직접 관여하지 않는다. 그저 PodTemplate을 해싱하여 ReplicaSet이라는 우직한 관리자를 낳고, 이 ReplicaSet이 "해당 해시값을 가진 Pod가 N개 존재하는가?"라는 단일 명제에만 집착하도록 만든다.

![](/images/2026/k8s-start/13.png)

Deployment의 진가는 세계가 변화(버전 업데이트)할 때 드러난다. 새 이미지가 배포되면 Deployment는 새로운 ReplicaSet을 창조하고, 낡은 ReplicaSet의 숨통을 서서히 조이면서 새로운 ReplicaSet의 세력을 넓힌다.
즉, Deployment의 본질은 단순히 "Pod 여러 개 띄우기"가 아니라, **"ReplicaSet 간의 트래픽 전환(Rollout)과 실패 시의 롤백(Undo)이라는 역사적 흐름을 통제하는 유한 상태 기계(State Machine)"**인 것이다.

### StatefulSet: **발 없는 새**

영화 <아비정전> 속 장국영(아비)이 남긴 유명한 독백이 있다.

```text
"세상에 발 없는 새가 있다더군. 늘 날아다니다가 지치면 바람 속에서 쉰대. 평생 딱 한 번 땅에 내려앉는데, 그건 바로 죽을 때지."
```

아비가 끊임없이 부유하며 누구와도 정착하지 못하는(가벼움) 진짜 이유는, 내면 깊숙한 곳에 억눌린 '친모에게 버림받았다는 지독한 상실감과 트라우마(무거움)' 때문이다. 겉으로 드러나는 극단적인 가벼움은 사실 스스로 감당하기 힘든 무거움이 존재하기에 선택한 도피처였다.

#### Deployment 아래의 무상태(Stateless) Pod들이 결벽증에 가까울 정도로 상태를 거부하고, 다른 노드나 디스크와의 끈끈한 의존성을 끊어내려 발버둥 치는 이유가 무엇일까?

데이터의 동기화, 트랜잭션의 무결성, 노드 장애 시의 데이터 유실... '상태(State)'라는 녀석은 조금만 어긋나도 분산 시스템 전체를 붕괴시킬 수 있는 끔찍한 중압감을 지닌다. 아비가 상처받지 않기 위해 관계를 거부했듯, 클라우드 네이티브 생태계는 장애라는 트라우마를 피하기 위해 상태를 철저히 외부(DB, 볼륨)로 밀어내고 스스로 **발 없는 새**가 되기를 택한 것이다.

하지만 세상에는 결코 무거움을 포기할 수 없는 존재들이 있다. StatefulSet은 데이터베이스처럼 반드시 땅(디스크)에 발을 딛고 자신의 정체성을 유지해야 하는 자들을 위한 요람이 된다.
StatefulSet은 각 Pod에 **예측 가능하고 영구적인 고유 식별자(예: web-0, web-1)**를 부여한다. 이들은 Headless Service와 결합하여 고유한 DNS를 얻고, 볼륨 템플릿(PVC)을 통해 자신의 식별자와 1:1로 묶인 영구적인 땅(디스크)을 분양받는다.

web-0 Pod가 죽어 다른 노드에서 예토전생하더라도, 쿠버네티스는 기어코 다시 'web-0'이라는 이름표를 쥐여주고 과거에 쓰던 디스크를 한 치의 오차 없이 찾아내어 연결해 준다. 가벼움을 잃은 대신, 불멸의 기억을 얻은 것이다.


### DaemonSet: 스케줄러를 우회하는 강박증

일반적인 컨트롤러들이 리소스 여유가 있는 노드를 이리저리 재면서 Pod를 배치하는 타협주의자라면, DaemonSet은 **"조건에 맞는 모든 노드에 반드시 1개씩"**이라는 무서운 강박증을 가진 통제광이다.
로그 수집기, 네트워크 플러그인(CNI) 등 클러스터의 근간을 지탱하는 인프라 에이전트들이 이 방식으로 배포된다. 한때는 스케줄러의 눈을 피해 노드에 직접 침투했지만, 이제는 정규 스케줄링을 타되 NodeAffinity를 극단적으로 강제하는 방식으로 그 끈질긴 생명력을 이어간다.

### Job과 CronJob: '죽음'을 목적으로 하는 제어 루프


![](/images/2026/k8s-start/14.png)

다른 컨트롤러들의 지상 과제가 프로세스의 영생(불사)이라면, Job 컨트롤러의 유일한 목적은 컨테이너 내부 프로세스의 **아름다운 죽음(Exit Code 0)**이다.
CronJob은 리눅스의 단순한 cron 데몬이 아니다. 정해진 시간마다 이 죽음을 수행할 Job '객체'를 etcd에 생성하며, 생성된 Job들의 성공과 실패 이력, 동시 실행 정책을 쿠버네티스 API라는 캔버스 위에 낱낱이 기록하고 통제한다.

## 3. 네트워킹과 트래픽: 가상 IP의 마법과 환상


### Service: kube-proxy가 엮어낸 거대한 착시

Service는 물리적인 로드밸런서 장비나 떠 있는 프로세스가 아니다. ClusterIP라는 가상의 IP는 리눅스의 물리적 네트워크 인터페이스 그 어디를 뒤져도 존재하지 않는 정교한 환상인 것이다.

![](/images/2026/k8s-start/6.png)

이 환상을 다루는 작업은 각 노드의 심연에 숨어있는 kube-proxy가 담당한다. 이 녀석은 API Server를 은밀히 감시하다가 Service와 그 뒤에 숨은 Pod IP들이 생성되면, 노드의 리눅스 커널(iptables 또는 IPVS)에 NAT(네트워크 주소 변환) 규칙을 촘촘한 거미줄처럼 깔아버린다.
애플리케이션이 허상의 Service IP로 패킷을 던지는 순간, 패킷이 노드의 문턱을 넘기도 전에 커널의 심층부에서 목적지 IP가 살아있는 Pod IP 중 하나로 낚아채듯 변환(DNAT)된다. 이것이 Service가 단 1바이트의 프록시 오버헤드 없이 완벽한 부하 분산을 해내는 소름 돋는 원리다.


### Headless Service: 착시를 걷어낸 클라이언트 사이드 디스커버리

clusterIP: None으로 선언된 Headless Service는 위에서 말한 iptables의 거미줄(가상 IP)을 아예 치지 않는다. 대신 쿠버네티스의 심장부인 CoreDNS에 Service 이름을 물어보면, 단일 IP 뒤에 숨겨진 연결된 모든 Pod의 적나라한 실제 IP 목록(A 레코드)을 뭉텅이로 던져준다.
이는 클라이언트가 직접 로드밸런싱 알고리즘을 쥐고 흔들거나, StatefulSet의 멤버들이 서로의 생사를 직접 확인해야 하는 P2P 아키텍처에서 환상 없이 날것의 네트워크를 마주하기 위해 쓰인다.

### Ingress와 Gateway API: L7 라우팅의 진화

![](/images/2026/k8s-start/17.png)

Service가 L4(IP/Port)의 맹목적인 배달부라면, L7(HTTP 경로, 헤더)의 섬세한 라우팅은 Ingress의 몫이다. Ingress 자체는 그저 "이렇게 길을 터달라"고 적어둔 종이 쪼가리(YAML)일 뿐이며, 이를 실제로 읽어 트래픽을 가르는 문지기는 NGINX 같은 Ingress Controller 프로세스다.

하지만 Ingress는 태생적으로 단순한 웹 트래픽만을 염두에 두었기에, 헤더 기반 카나리 배포나 gRPC 같은 현대의 복잡다단한 요구를 담아내기엔 그릇이 너무 작았다. 이를 부수고 등장한 Gateway API는 인프라 관리자(GatewayClass, Gateway)와 앱 개발자(HTTPRoute)의 권한과 책임을 객체 단위로 철저히 분리시켰다. 

## 4. 설정과 스토리지: 영혼과 육신, 그리고 땅의 디커플링


### ConfigMap과 Secret: 런타임 환경의 분리

도커 이미지가 지켜야 할 가장 근본적인 규칙은 '불변성(Immutability)'이다. 개발자의 랩탑과 프로덕션 서버에서 이미지가 달라진다는 것은 이 세계의 붕괴를 뜻한다. ConfigMap과 Secret은 이 굳건한 이미지(육신)에 환경마다 다른 설정과 자격 증명(영혼)을 외부에서 공급할 수 있게 한 현명한 전략이다.

환경변수(Env)로 꽂힌 값은 컨테이너가 숨을 쉴 때(프로세스 시작) 단 한 번 폐부로 들어오기 때문에 변경 시 Pod의 죽음(재시작)이 불가피하다. 하지만 볼륨(Volume)으로 마운트된 설정 파일은 kubelet이 주기적으로 새로운 값을 동기화해 주므로, 애플리케이션이 변화를 감지(Hot Reload)할 수만 있다면 무중단으로 영혼을 갈아 끼우는 것이 가능하다.
(주의: Secret은 etcd에 그저 base64로 얕게 인코딩되어 널브러져 있을 뿐, 절대 암호화된 상태가 아니다. 진정한 비밀 유지를 위해서는 AWS KMS나 Vault 같은 외부의 금고와 결탁해야만 한다.)

### PV, PVC, 그리고 StorageClass: 땅(스토리지)의 추상화

과거에는 스토리지를 쓰려면 개발자가 스토리지 서버의 물리적 IP와 디스크 경로를 Pod 명세에 하드코딩해야 했다. 이는 쿠버네티스의 가장 큰 무기인 '어디서든 뜰 수 있다'는 이식성을 심각하게 훼손하는 족쇄였다. 

![](/images/2026/k8s-start/15.png)
이 족쇄를 부수기 위해 탄생한 것이 요청(PVC)과 자원(PV)의 철저한 분리이다.

    PVC (PersistentVolumeClaim): "나는 내부가 어떻게 생겼든 상관없으니 읽기/쓰기가 빠른 10GB짜리 땅(디스크)이 필요하다."

    PV (PersistentVolume): 인프라 관리자가 생성해둔 실제 물리적 스토리지 자원.

하지만 관리자가 매번 요청이 올라올 때마다 수동으로 땅을 파서 PV를 바치는 것은 끔찍한 노동이다. 여기서 StorageClass와 동적 프로비저닝(Dynamic Provisioning)이라는 기법이 나타난다.

개발자가 PVC에 storageClassName: aws-ebs-gp3라는 명령을 사용하면, CSI 드라이버가 AWS API를 직접 호출해 클라우드의 땅을 즉석에서 도려내고, 이를 PV로 빚어내어 PVC와 강제로 결속(Binding)시킨다.
개발자는 물리 인프라의 흙먼지를 뒤집어쓰지 않고도, 오직 선언만으로 영구적인 땅을 얻어내는 결과를 얻는 것이다.

## 5. Pod 내부 런타임 설계: 환상을 깨는 리눅스 커널의 현실

![](/images/2026/k8s-start/20.png) 

가장 먼저 머릿속에서 도려내야 할 착각은 "Pod가 일종의 작은 가상 머신(VM)처럼 동작한다"는 환상이다. 리눅스 커널의 관점에서 Pod라는 이름의 물리적 프로세스는 결코 존재하지 않는다. Pod는 그저 동일한 리눅스 네임스페이스(Network, IPC, UTS)라는 보이지 않는 울타리를 공유하고, cgroups라는 동일한 목줄(자원 할당량)에 묶여 있는 독립된 컨테이너 프로세스들의 논리적인 환영일 뿐이다

그럼 이 환영을 조율하는 실질적인 지배자는 누구일까?
바로 kubelet 이다.

### Init Container와 Sidecar: 네임스페이스 공유의 기술

![](/images/2026/k8s-start/16.png)

kubelet이 Pod라는 무대를 세울 때 가장 먼저 띄우는 것은 pause라는 텅 빈 껍데기 인프라 컨테이너다. 오직 IP 주소라는 네트워크 네임스페이스을 기억하기 위함이다.
그다음 해당 네임스페이스 아래로 Init Container들이 차례로 입장한다. 이들은 메인 앱이 밟고 설 땅을 다지고, 권한을 부여하고, 스키마를 붓고는 미련 없이 스스로 목숨을 끊는다(Exit 0). 이들이 모두 죽었을 때, 비로소 완성되는 것이다.

반면 Sidecar Container는 메인 앱과 생사를 함께하는 그림자다. 이들은 pause 컨테이너가 만들어둔 동일한 네트워크 울타리 안으로 합류한다. 덕분에 앱 컨테이너와 Envoy 프록시 같은 사이드카는 서로를 localhost라는 가장 내밀한 이름으로 부를 수 있고, 빈 디렉터리(emptyDir)를 통해 심장 박동처럼 데이터를 실시간으로 주고받을 수 있다.

### Probe: kubelet의 생사 여탈권

Probe는 멀리 떨어진 컨트롤 플레인이 아니라, 노드의 최전선에 선 kubelet이 직접 들이대는 날카로운 메스(Health Check)다.

    startupProbe: 무거운 Spring Boot나 JVM 앱이 기지개를 켤 때, livenessProbe가 이를 장애로 오진하여 성급하게 목을 치는 참사를 막는 **유예 기간의 방패**다. 이 방패가 걷히기 전까지 다른 메스는 모두 잠잠하다.

    livenessProbe: 이 검사에 실패하면 kubelet은 지체 없이 컨테이너의 뇌수(PID 1)에 SIGTERM 시그널을 박아 넣어 프로세스를 박살 내고 새로 띄운다. 교착 상태(Deadlock)에 빠진 좀비를 처리하는 물리적인 철퇴다. 

    readinessProbe: 이는 실패하더라도 프로세스를 살려두되, API Server에 즉각 보고하여 Service의 iptables 라우팅 타겟 목록에서 이 Pod의 IP를 파내버린다. "심장은 뛰지만 트래픽을 받을 정신은 없는" 상태를 사회망으로부터 완벽히 격리하는 섬세한 장치다.

### Lifecycle Hook (PreStop): 우아한 죽음을 맞이할 준비 시간

Pod가 삭제될 때 kubelet은 무자비하게 단두대의 칼날(SIGKILL)을 바로 내리치지 않는다. API Server의 엔드포인트에서 IP를 파내는 비동기 작업이 시작됨과 동시에, kubelet은 컨테이너 내부에 PreStop 훅을 던져준다.

이것은 치명적인 SIGTERM 시그널이 도달하기 전, 프로세스에 허락된 마지막 유언의 시간이다. 네트워크의 한계로 iptables 갱신이 채 끝나기 전 밀려들어 오는 잔여 트래픽을 처리하거나, 물고 있던 DB 커넥션을 안전하게 내려놓을(Draining) 생을 정리할 마지막 순간인 것이다. 이 PreStop과 terminationGracePeriodSeconds의 호흡을 맞추지 못하면, 배포가 일어날 때마다 유저들은 502 Bad Gateway라는 에러에 또 서버 터졌냐고 빗발치는 문의가 들어올 것이다.

## 6. 리소스와 스케줄링: 장부의 숫자와 커널의 철퇴

![](/images/2026/k8s-start/11.png)

### Requests와 Limits: 스케줄러와 커널의 엇갈린 시선

이 두 설정이 바라보는 세계관은 완전히 다르다.

![](/images/2026/k8s-start/7.png)

    requests (요청량): kube-scheduler가 노드에 Pod를 욱여넣을 때(Bin-packing) 참고하는 '희망찬 장부 상의 숫자'다. 스케줄러는 노드의 물리적인 실시간 잔여 자원을 보지 않는다. 그저 현재 할당된 Pod들의 requests 총합만을 계산하여 빈자리를 가늠할 뿐이다.

    limits (제한량): 배치가 끝난 뒤, 노드의 kubelet이 컨테이너 런타임을 통해 리눅스 커널의 cgroups 에 적용하는 '냉혹한 물리적 하드 리미트'다. CPU limit을 넘기면 커널의 공평성 스케줄러(CFS)가 프로세스의 멱살을 잡아 느리게 만들고(Throttling), Memory limit을 넘기는 순간 커널의 암살자(OOM Killer)가 깨어나 프로세스의 숨통을 끊어버린다(Kill).

### QoS Class와 Eviction (퇴거): 가라앉는 배에서의 생존 계급

노드의 메모리가 고갈되어 운영체제 전체가 멈춰 설 위기에 처하면, kubelet은 뱃머리를 돌리기 위해 자신이 품었던 Pod들을 바다로 밀어 던져야(Eviction) 한다. 이때 리눅스 커널의 oom_score_adj 값을 조작하여 생존과 죽음의 순위를 결정짓는 자가 바로 QoS Class다.

![](/images/2026/k8s-start/12.png)

    Guaranteed (Request = Limit): 한 치의 자원 오차도 허용치 않는 귀족. 배가 완전히 침몰하기 전까지 가장 마지막까지 살아남는 VIP다.

    Burstable (Request < Limit): 약속된 자원 이상으로 메모리를 탐욕스럽게 끌어다 쓰는 평민 계급. 압박이 심해지면 우선 타겟이 된다.

    BestEffort (설정 없음): 아무런 자원 설정도 하지 않은 자들. 노드에 미세한 압박만 와도 가장 먼저 바다로 버려지는 비참한 0순위 희생양이다. "대충 돌아가겠지"라는 안일함이 대규모 연쇄 장애라는 참극을 부르는 가장 큰 이유다.

### Taint/Toleration과 Affinity: 스케줄러의 2단계 필터링 체계

스케줄러가 Pod를 안착시킬 노드를 고르는 과정은 철저히 '배제(Filtering)'와 '유혹(Scoring)'의 이중 구조로 이루어진다.

![](/images/2026/k8s-start/10.png)

    Taint & Toleration (필터링의 배제): 노드가 방어막(Taint)을 두른다. "여기는 성스러운 GPU의 땅이니, 이 특별한 면죄부(Toleration)가 없는 Pod는 절대 발을 들이지 마라." 철저하고 물리적인 배척 장치다.

![](/images/2026/k8s-start/9.png)

    Node/Pod Affinity (스코어링의 유혹): "이왕이면 SSD가 달린 노드(Node Affinity)로 가고 싶다", "이왕이면 내 곁에 Redis 캐시 Pod가 있는 따뜻한 곳(Pod Affinity)으로 배포해 다오." 이는 스케줄러가 노드들의 점수를 매길 때 마음을 흔드는 논리적이고 부드러운 중력 장치다.

## 7. 보안과 멀티테넌시: API Server의 문지기들

### ServiceAccount: 컨테이너의 신분증 (JWT)

애플리케이션이 쿠버네티스의 심장(API Server)에 접근하거나 클라우드 자원(AWS S3 등)를 부리려면 자신이 누구인지 철저히 증명해야 한다. 이때 쥐여주는 통행증이 바로 ServiceAccount다.
이 통행증의 물리적 실체는 API Server가 암호화 키로 직접 서명한 **JWT(JSON Web Token)**다. kubelet은 이 토큰을 노드의 메모리 기반 파일시스템(tmpfs)인 Projected Volume으로 빚어내어, 컨테이너 내부 /var/run/secrets/... 경로에 저장한다. 프로세스는 이 토큰을 꺼내어 API Server의 문지기에게 제출하고 인가(Authorization)를 얻어낸다.


### SecurityContext: 왕관의 박탈과 리눅스 Capabilities

컨테이너 안에서 root 권한을 쥐고 흔드는 프로세스는, 단지 네임스페이스라는 얇은 유리벽에 갇혀 있을 뿐 사실 호스트 리눅스 머신에서도 동일한 root의 왕관을 쓴 폭군이다. 해커가 이 유리벽을 깨는 순간(Container Breakout), 노드 전체가 그들의 식민지로 전락할 수 있다는 위험이 존재한다.

SecurityContext는 이 폭거를 커널 레벨에서 짓누른다. 
runAsNonRoot: true로 프로세스의 권한을 평민(일반 유저)으로 강제 강등시키거나,
리눅스 커널의 세밀한 권한 제어(Capabilities)를 통해 네트워크 설정 권한(CAP_NET_ADMIN) 같은 위험한 권력을 명시적으로 모조리 뽑아버린다(Drop).

## 8. 확장과 플랫폼화: 무한 루프의 연금술


### CRD와 Operator: 선언적 제어 루프의 무한 확장

쿠버네티스가 도커 스웜을 짓밟고 현대 클라우드의 지배자가 된 이유는, 이것이 고정된 컨테이너 실행기가 아니라 모든 것을 집어삼킬 수 있는 'API 확장 프레임워크'이기 때문일 것이다.
(도커 스웝은 다루지 않을 예정이다. 도커 자체에 이미 도커 스웜은 탑재되어 있으니 직접 사용해봐도 좋다.)

    CRD (Custom Resource Definition): etcd라는 거대한 뇌에 새로운 개념(데이터 스키마)을 주입하는 작업이다. Prometheus나 KafkaCluster 같은 미지의 YAML 덩어리를 API Server가 인식하고 기록하게 만든다. 하지만 CRD는 생명력이 없는 껍데기에 불과하다.

    Operator (Custom Controller): 이 껍데기에 생명을 주는 Control Loop 이다. 이 프로세스는 API Server의 변화를 실시간으로 감시(Informer 메커니즘)하다가, 인간이 KafkaCluster YAML을 툭 던져놓으면 그 즉시 깨어나 내부적으로 StatefulSet, ConfigMap, Service를 설계도대로 찍어내고 백업까지 자동으로 수행한다.
   
    즉, **인프라 고인물의 도메인 지식과 눈물겨운 운영 런북(Runbook)을 코드로 박제하여 쿠버네티스의 뇌 안에 영구히 상주시키는 것**이 Operator 패턴의 숙명이라 할 수 있겠다.
    

## 9. 실무의 사유: 의도(Intent)의 위임

지금까지 살펴본 모든 컴포넌트의 물리적 동작은 결국 **명령(Imperative)하지 말고, 선언(Declarative)하라** 을 위한 빌드업이었다.
인간이 개입한 그 모든 발자취는, 어차피 kubelet이나 상위 컨트롤러가 원래 선언된 진리(YAML)로 가차 없이 덮어써 버리기에 지양되어야 한다.

    Pod가 이유 없이 자꾸 죽는가? 로그를 보며 프로세스를 탓하기 전에, livenessProbe의 잣대가 현실의 비루함을 담아내지 못한 건 아닌지, 극한의 limits 설정 탓에 커널의 OOM Killer가 조용히 암살을 집행하고 간 것은 아닌지 확인해야 한다.

    트래픽이 목적지를 잃고 헤매는가? Pod 내부에서 핑(Ping)을 날리며 허공에 삽질할 게 아니라, kube-proxy가 노드의 iptables에 핏줄(엔드포인트)을 제대로 연결했는지, readinessProbe가 아직 준비되지 않은 Pod를 철저히 격리하고 있는지 그 위계 질서를 의심해야 한다.

결국 쿠버네티스를 다루는 본질은, "내가 염원하는 세계의 최종 상태(Desired State)를 얼마나 정교하고 빈틈없는 API 객체의 조합으로 번역해 etcd라는 신전 앞에 바칠 수 있는가"에 달린 것 같다. 인류의 놀라운 기술 발전은 기록으로 시작되었고, 그 본질은 바뀌지 않았으니까.
