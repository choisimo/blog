처음 Java를 배울 때 가장 헷갈리는 순간은 대개 어려운 문법에서 오지 않는다. 오히려 너무 익숙해 보이는 단어들에서 온다. 클래스, 객체, 상속, 오버라이딩, 오버로딩, static. 이름만 보면 서로 분리된 개념처럼 보인다. 그래서 처음에는 각각의 정의를 외우려고 한다.

오버로딩은 같은 이름의 메서드를 여러 개 만드는 것.
오버라이딩은 부모의 메서드를 자식이 다시 정의하는 것.
상속은 부모의 기능을 물려받는 것.
static은 객체 없이 클래스에 붙는 것.

틀린 설명은 아니다. 다만 이 설명만으로는 실제 코드가 왜 그런 결과를 내는지 끝까지 따라가기 어렵다. 특히 시험 문제나 면접 문제에서 Java 코드는 일부러 사람의 직관을 흔든다. 사람은 코드를 위에서 아래로 읽지만, Java 런타임은 그렇게만 움직이지 않는다. 어떤 결정은 컴파일 시점에 끝나고, 어떤 결정은 실행 시점까지 미뤄진다. 어떤 이름은 변수의 타입을 따라가고, 어떤 호출은 실제 객체의 타입을 따라간다.

이 지점에서 Java는 단순한 객체지향 언어가 아니라, “선언된 타입”과 “실제 객체” 사이의 긴장을 관리하는 실행 시스템이 된다.

## Java를 문법 목록으로 보면 왜 헷갈리는가

Java를 처음 배우는 사람은 보통 클래스 중심으로 생각한다.

```java
class A {
    void hello() {
        System.out.println("A");
    }
}

class B extends A {
    void hello() {
        System.out.println("B");
    }
}
```

이 코드는 어렵지 않아 보인다. `B`는 `A`를 상속했고, `hello()`를 다시 정의했다. 그래서 `B` 객체에서 `hello()`를 부르면 `"B"`가 출력된다. 여기까지는 자연스럽다.

문제는 다음 코드에서 시작된다.

```java
A x = new B();
x.hello();
```

겉으로 보면 변수 `x`의 타입은 `A`다. 그런데 실제 출력은 `"B"`다. 여기서 많은 사람이 첫 번째 균열을 느낀다.

“변수 타입은 A인데 왜 B의 메서드가 실행되지?”

이 질문은 좋은 질문이다. Java의 다형성(Polymorphism)은 바로 이 불편함 위에 서 있다. 변수는 `A`라는 입구를 제공하지만, 실제로 문을 열고 들어가면 그 안에는 `B` 객체가 있다. Java는 메서드 오버라이딩 호출에서 선언된 변수 타입만 보지 않는다. 실행 시점에 실제 객체가 무엇인지 확인하고, 그 객체에 맞는 메서드를 호출한다.

이것을 단순히 “동적 바인딩(Dynamic Binding)”이라고 외우면 또다시 이름만 남는다. 중요한 것은 상태가 두 층으로 나뉜다는 점이다.

| 구분       | 의미                        |
| -------- | ------------------------- |
| 참조 변수 타입 | 컴파일러가 허용할 수 있는 메서드 범위     |
| 실제 객체 타입 | 실행 시점에 실제로 선택되는 오버라이딩 메서드 |

`A x = new B();`는 `x`가 `A`라고 말하는 코드가 아니다. 더 정확히는 `x`라는 참조 변수가 `A` 타입의 인터페이스로 `B` 객체를 가리킨다는 뜻이다. 변수 타입은 접근 가능한 문을 정하고, 실제 객체 타입은 문 뒤에서 어떤 동작이 일어날지를 정한다.

이 둘을 하나로 생각하면 Java 문제는 갑자기 이상해진다.

## 오버라이딩은 “이름 재사용”이 아니라 실행 책임의 교체다

오버라이딩(Overriding)을 “부모 메서드를 자식 메서드로 덮어쓴다”고 설명하는 경우가 많다. 이 표현은 어느 정도 맞지만, 실제 구조를 조금 흐린다. 부모 클래스의 메서드가 파일에서 사라지는 것은 아니다. 부모의 코드도 여전히 존재한다. 다만 실행 시점에 어떤 객체를 대상으로 호출했는지에 따라 책임을 맡는 메서드가 달라진다.

```java
class Animal {
    void sound() {
        System.out.println("some sound");
    }
}

class Dog extends Animal {
    @Override
    void sound() {
        System.out.println("bark");
    }
}
```

```java
Animal a = new Dog();
a.sound();
```

여기서 `a.sound()`는 표면적으로 `Animal` 타입 변수에서 호출된다. 하지만 실제 객체는 `Dog`다. 그래서 호출은 `Dog.sound()`로 이동한다.

이 장면을 상태 전이로 보면 더 분명해진다.

```text
현재 상태:
참조 변수 a
  declared type = Animal
  actual object = Dog

입력:
a.sound()

전이:
컴파일 시점: Animal에 sound()가 있는지 확인
실행 시점: actual object인 Dog에서 override된 sound() 선택

최종 상태:
Dog.sound() 실행
```

여기서 유지되는 불변식은 이것이다.

참조 변수의 선언 타입은 “호출 가능한 메서드의 목록”을 제한한다.
실제 객체 타입은 “오버라이딩된 메서드의 최종 실행 대상”을 결정한다.

그래서 다음 코드는 실패한다.

```java
class Animal {
    void sound() {}
}

class Dog extends Animal {
    void sound() {}
    void wagTail() {}
}

Animal a = new Dog();
a.wagTail(); // 컴파일 에러
```

실제 객체는 `Dog`이지만, 변수 `a`의 선언 타입은 `Animal`이다. 컴파일러는 `a`를 `Animal` 입구로만 본다. `Dog` 안에 `wagTail()`이 있다는 사실은 실행 시점에는 맞지만, 컴파일 시점의 접근 규칙을 통과하지 못한다.

이것이 Java의 중요한 구조다. 실행 시점의 자유는 컴파일 시점의 약속을 넘지 못한다.

## 오버로딩은 실행 시점이 아니라 컴파일 시점에 더 가깝다

오버로딩(Overloading)은 오버라이딩과 이름이 비슷해서 자주 섞인다. 둘 다 같은 이름의 메서드가 여러 개 등장한다. 하지만 두 개념은 결정되는 시점이 다르다.

오버라이딩은 실제 객체 타입을 따라 실행 시점에 결정된다.
오버로딩은 전달 인자의 선언 타입을 보고 컴파일 시점에 후보가 정해진다.

```java
class Printer {
    void print(Object x) {
        System.out.println("Object");
    }

    void print(String x) {
        System.out.println("String");
    }
}
```

```java
Printer p = new Printer();
p.print("hello");
```

이 경우 `"hello"`는 문자열 리터럴이고, `String` 타입이다. 따라서 `print(String x)`가 선택된다. 여기까지는 직관적이다.

하지만 다음 코드는 다르다.

```java
Object value = "hello";
p.print(value);
```

실제 객체는 문자열이다. 하지만 변수 `value`의 선언 타입은 `Object`다. 오버로딩 선택은 실제 객체가 아니라 컴파일 시점의 인자 타입에 강하게 묶인다. 그래서 `print(Object x)`가 선택된다.

이 차이가 Java 문제를 어렵게 만든다. 같은 메서드 이름을 보고도 어떤 것은 실제 객체를 따라가고, 어떤 것은 선언 타입을 따라간다.

```text
오버라이딩:
호출 대상 객체의 실제 타입을 본다.

오버로딩:
인자로 넘긴 표현식의 컴파일 시점 타입을 본다.
```

비유하자면 오버라이딩은 “실제로 누가 그 자리에 앉아 있는가”를 본다. 반면 오버로딩은 “접수 서류에 어떤 직책으로 적혀 있는가”를 본다. 실제 사람이 더 많은 능력을 갖고 있어도, 접수 창구는 서류에 적힌 타입으로 분류한다.

이 비유를 코드로 되돌리면, Java에서 이름이 같다고 해서 같은 규칙으로 선택되는 것이 아니다. 선택 시점이 다르면 결과도 달라진다.

## 헷갈리는 문제는 대부분 이 두 규칙이 겹칠 때 나온다

다음 코드를 보자.

```java
class A {
    String f(Object x) {
        return "1";
    }

    String g() {
        return f("a");
    }
}

class B extends A {
    String f(Object x) {
        return "2";
    }

    String f(String x) {
        return "3";
    }
}

public class Main {
    public static void main(String[] args) {
        A a = new B();
        System.out.println(a.g());
    }
}
```

처음 보면 `a`는 `new B()`를 가리키고 있다. 그러면 `B`의 메서드가 실행될 것 같고, `"3"`이 출력될 것처럼 보인다. 하지만 실제 출력은 `"2"`다.

이 문제를 외우려고 하면 헷갈린다. 대신 상태를 나누어 보면 된다.

```text
현재 상태:
a의 선언 타입 = A
a의 실제 객체 = B

입력:
a.g()

1단계:
g()는 A에만 정의되어 있다.
B는 g()를 오버라이딩하지 않았다.
따라서 실행되는 메서드 본문은 A.g()다.

2단계:
A.g() 내부에서 f("a")를 호출한다.

3단계:
A.g()가 컴파일될 당시,
A 클래스 안에는 f(Object)만 보인다.
B의 f(String)는 A.g()의 오버로딩 후보가 아니다.

4단계:
따라서 A.g() 내부의 f("a")는 f(Object) 호출로 컴파일된다.

5단계:
실행 시점에 f(Object)는 오버라이딩 대상이다.
실제 객체는 B이므로 B.f(Object)가 실행된다.

최종 출력:
"2"
```

이 코드는 Java가 왜 문법 암기만으로 해결되지 않는지를 잘 보여준다. 표면적으로는 `f("a")`라서 `f(String)`이 선택될 것 같다. 하지만 그 호출이 어디에 적혀 있는지가 중요하다. `f("a")`는 `A.g()` 내부에 있다. `A.g()`가 컴파일될 때 `A`는 `B.f(String)`을 모른다.

여기서 결정적인 분기점은 두 개다.

| 단계       | 질문                                   | 결과            |
| -------- | ------------------------------------ | ------------- |
| 오버로딩 선택  | `A.g()` 안에서 `f("a")`는 어떤 시그니처로 묶이는가? | `f(Object)`   |
| 오버라이딩 선택 | 실행 시점에 `f(Object)`의 실제 구현은 누구인가?     | `B.f(Object)` |

그래서 `"3"`이 아니라 `"2"`가 나온다.

`B.f(String)`은 존재한다. 하지만 이 호출 흐름에서는 선택될 기회를 얻지 못한다. 존재하는 코드와 도달 가능한 코드는 다르다.

## static은 객체의 세계에서 한 발 떨어져 있다

`static`도 처음에는 간단해 보인다. 객체를 만들지 않고 클래스 이름으로 접근할 수 있는 멤버. 이 정도 설명이면 기본 예제는 풀 수 있다.

```java
class Counter {
    static int count = 0;

    Counter() {
        count++;
    }
}
```

```java
new Counter();
new Counter();

System.out.println(Counter.count); // 2
```

`count`는 객체마다 따로 있는 값이 아니라 클래스에 붙어 있는 값이다. 인스턴스가 여러 개 생겨도 `count`는 하나다.

하지만 static의 본질은 “편리한 전역 변수”가 아니다. static은 객체 단위로 분리되어야 할 상태와 클래스 단위로 공유되어야 할 상태를 나누는 경계다. 이 경계를 잘못 잡으면 작은 코드에서는 편해 보이지만, 규모가 커질수록 상태가 서로 엉킨다.

```java
class UserContext {
    static String currentUser;
}
```

처음에는 이 코드가 편해 보일 수 있다. 어디서든 `UserContext.currentUser`로 현재 사용자를 읽으면 된다. 하지만 여러 요청이 동시에 들어오는 서버에서는 이 구조가 위험해진다. 사용자 A의 요청이 값을 넣은 직후, 사용자 B의 요청이 같은 static 필드를 덮어쓸 수 있다.

```text
요청 A: currentUser = "alice"
요청 B: currentUser = "bob"
요청 A: currentUser 읽기 → "bob"
```

작은 프로그램에서는 문제가 없어 보였던 선택이 동시성 환경에서는 병목이 아니라 오류가 된다. static은 상태를 공유한다. 공유 상태는 항상 질문을 요구한다.

누가 바꾸는가.
언제 바뀌는가.
동시에 접근하면 어떤 값이 보이는가.
테스트가 끝난 뒤에도 값이 남아 있지는 않은가.

static을 쓰면 안 된다는 말이 아니다. 상수, 순수 함수, 객체 상태와 무관한 유틸리티에는 static이 자연스럽다.

```java
class MathUtil {
    static int square(int x) {
        return x * x;
    }
}
```

이 코드는 외부 상태를 바꾸지 않는다. 입력이 같으면 출력도 같다. 이런 static은 비교적 안전하다. 문제는 static이 “공유되는 변경 가능한 상태”를 들고 있을 때다. 그 순간 static은 단순한 문법이 아니라 전역 상태 관리 문제가 된다.

## 상속은 재사용 도구이기 전에 대체 가능성의 약속이다

상속(Inheritance)을 “부모 코드를 물려받는 기능”으로만 이해하면 사용법은 쉽다. 공통 필드를 부모에 올리고, 자식 클래스에서 재사용하면 된다.

```java
class User {
    String name;
}

class AdminUser extends User {
    boolean canDelete;
}
```

하지만 상속의 더 깊은 의미는 코드 복사 절약이 아니다. 상속은 “자식 객체를 부모 타입으로 다뤄도 시스템이 깨지지 않는다”는 약속이다.

```java
void printName(User user) {
    System.out.println(user.name);
}
```

이 메서드는 `User`를 받는다. 그런데 `AdminUser`도 넘길 수 있다.

```java
printName(new AdminUser());
```

왜 가능한가. `AdminUser`는 적어도 `User`로 취급될 수 있어야 하기 때문이다. 이 약속이 깨지면 상속은 재사용이 아니라 위장에 가까워진다.

예를 들어 `Bird`와 `Penguin`을 생각해볼 수 있다.

```java
class Bird {
    void fly() {
        System.out.println("fly");
    }
}

class Penguin extends Bird {
    @Override
    void fly() {
        throw new UnsupportedOperationException();
    }
}
```

현실 세계에서 펭귄은 새다. 그래서 `Penguin extends Bird`가 자연스러워 보인다. 하지만 코드 세계에서는 `Bird`가 `fly()`를 가진 타입으로 정의되어 있다. 이 경우 `Penguin`은 `Bird`의 동작 계약을 깨뜨린다.

처음에는 현실의 분류가 합리적으로 보인다. 하지만 소프트웨어의 타입은 생물학 도감이 아니다. 타입은 프로그램 안에서 허용되는 동작의 계약이다.

이 차이를 놓치면 상속은 구조를 단순하게 만드는 대신 예외와 조건문을 늘린다.

```java
if (bird instanceof Penguin) {
    // 날지 못함
} else {
    bird.fly();
}
```

이런 코드가 늘어난다는 것은 상속 구조가 실제 동작 모델과 맞지 않는다는 신호일 수 있다. 객체지향은 현실을 그대로 코드에 옮기는 것이 아니다. 프로그램이 유지해야 하는 불변식에 맞게 세계를 다시 자르는 일에 가깝다.

## Java의 추상화는 비용을 없애지 않고 뒤로 미룬다

Java는 많은 것을 감춰준다. 포인터 연산을 직접 하지 않아도 된다. 메모리 해제도 직접 호출하지 않는다. JVM(Java Virtual Machine)은 운영체제 위에 또 하나의 실행 환경을 만들고, 바이트코드(Bytecode)는 여러 플랫폼에서 비슷하게 실행될 수 있게 한다.

이 추상화 덕분에 Java는 오래 살아남았다. 하지만 추상화는 비용을 삭제하지 않는다. 단지 개발자가 매 순간 보지 않아도 되게 뒤로 밀어둘 뿐이다.

객체를 계속 만들면 언젠가 가비지 컬렉션(Garbage Collection)이 개입한다.
스레드를 쉽게 만들 수 있어도 컨텍스트 스위칭(Context Switching) 비용은 사라지지 않는다.
컬렉션 프레임워크(Collection Framework)가 편리해도 내부 배열 복사, 해시 충돌, 박싱(Boxing) 비용은 남아 있다.
JVM이 플랫폼 차이를 감춰도, 실제 운영 환경의 메모리 제한과 CPU 사용률은 감춰지지 않는다.

작은 규모에서는 이런 비용이 거의 보이지 않는다. 그래서 처음에는 Java의 추상화가 공짜처럼 느껴진다. 문제는 상태가 누적되고, 요청 수가 늘고, 객체 생명주기가 복잡해지는 순간이다. 그때부터 감춰진 비용이 로그, 지연 시간, GC pause, 메모리 사용량, 스레드 덤프 같은 형태로 표면에 올라온다.

예를 들어 다음 코드는 문법적으로는 매우 간단하다.

```java
List<Integer> numbers = new ArrayList<>();

for (int i = 0; i < 1_000_000; i++) {
    numbers.add(i);
}
```

하지만 내부에서는 `int`가 `Integer`로 박싱될 수 있고, `ArrayList`의 내부 배열이 부족해질 때마다 더 큰 배열로 교체되며 기존 요소가 복사된다. 겉으로는 `add` 한 줄이지만, 아래에서는 메모리 할당과 객체 생성, 배열 확장, 이후 GC 부담이 함께 움직인다.

이것이 Java를 배울 때 중요한 관점이다. 문법은 표면의 API다. 성능과 장애는 대개 그 아래 계층에서 발생한다.

## Java를 잘 이해한다는 것은 이름이 아니라 결정 시점을 아는 것이다

Java 문제를 풀 때 가장 먼저 봐야 할 것은 “이 문법의 이름이 무엇인가”가 아니다. 더 중요한 질문은 이것이다.

이 결정은 언제 내려지는가?

컴파일 시점인가.
실행 시점인가.
변수의 선언 타입을 보는가.
실제 객체 타입을 보는가.
클래스에 붙은 상태인가.
객체마다 분리된 상태인가.
공유되는 값인가.
요청마다 달라져야 하는 값인가.

이 질문을 기준으로 보면 헷갈리던 개념들이 조금씩 제자리를 찾는다.

| 개념     | 표면 설명          | 실제로 봐야 할 것          |
| ------ | -------------- | ------------------- |
| 오버로딩   | 같은 이름, 다른 매개변수 | 컴파일 시점의 인자 타입       |
| 오버라이딩  | 부모 메서드 재정의     | 실행 시점의 실제 객체 타입     |
| 상속     | 부모 기능 재사용      | 부모 타입으로 대체 가능한가     |
| static | 객체 없이 접근       | 클래스 단위 공유 상태인가      |
| 다형성    | 여러 형태          | 선언 타입과 실제 객체 타입의 분리 |
| JVM    | Java 실행 환경     | 추상화가 감춘 런타임 비용      |

여기서 `add`, `replace`, `override`, `dispatch`, `commit` 같은 동작은 단순한 단어가 아니다. Java 런타임은 코드의 각 지점에서 후보를 만들고, 제거하고, 선택하고, 최종 실행 대상을 commit한다. 오버로딩 후보는 컴파일러가 먼저 좁힌다. 오버라이딩 대상은 런타임이 실제 객체를 보고 다시 결정한다. static 필드는 객체 생성과 무관하게 클래스 로딩 단위에서 유지된다.

이 흐름을 보지 못하면 Java는 예외적인 규칙들의 묶음처럼 보인다. 반대로 결정 시점을 따라가면, 많은 문제는 하나의 상태 전이로 정리된다.

```text
현재 상태:
선언 타입
실제 객체 타입
메서드 후보 목록
클래스 공유 상태
인스턴스 개별 상태

입력:
메서드 호출
객체 생성
필드 접근
타입 변환

전이:
후보 선택
접근 가능성 검사
동적 디스패치
상태 변경
결과 반환

최종 상태:
어떤 코드가 실행되었는가
어떤 값이 바뀌었는가
어떤 타입 약속이 유지되었는가
```

## 한계와 트레이드오프

Java의 구조가 항상 우아한 것은 아니다. 강한 타입 시스템은 많은 오류를 컴파일 시점에 잡아주지만, 때로는 코드가 장황해진다. 객체지향은 책임을 나누는 데 유리하지만, 상속을 잘못 쓰면 오히려 결합도가 높아진다. JVM은 운영체제 차이를 줄여주지만, 그만큼 런타임 계층을 하나 더 이해해야 한다.

static은 편리하지만 공유 상태를 만들 수 있다.
상속은 재사용을 돕지만 잘못된 계층 구조를 굳힐 수 있다.
다형성은 확장을 쉽게 만들지만 실제 호출 대상을 추적하기 어렵게 만들 수 있다.
가비지 컬렉션은 메모리 해제를 자동화하지만 멈춤 시간과 튜닝 문제를 남긴다.

완전한 해법은 없다. Java를 잘 쓴다는 것은 모든 기능을 피하는 것이 아니라, 각 기능이 어떤 비용을 뒤로 미루는지 알고 선택하는 것이다. static을 쓰되 변경 가능한 공유 상태를 경계하고, 상속을 쓰되 대체 가능성을 확인하고, 다형성을 쓰되 호출 결정 시점을 이해해야 한다.

도구를 무조건 믿는 것도 위험하고, 무조건 피하는 것도 답은 아니다. 중요한 것은 그 도구가 어느 계층의 복잡도를 줄이고, 어느 계층의 비용을 새로 만드는지 보는 일이다.

## 정리: Java는 문법이 아니라 실행 시점의 약속이다

Java를 단순히 클래스와 객체의 언어로만 보면 많은 문제가 암기 대상이 된다. 오버로딩은 무엇이고, 오버라이딩은 무엇이고, static은 무엇인지 따로 외워야 한다. 하지만 한 계층 아래로 내려가면 질문은 더 단순해진다.

이 코드는 어떤 타입으로 컴파일되는가.
실행 시점의 실제 객체는 무엇인가.
메서드 후보는 언제 결정되는가.
상태는 객체마다 분리되는가, 클래스 전체에 공유되는가.
이 구조는 작은 규모에서만 편한가, 커졌을 때도 같은 불변식을 유지하는가.

Java는 우리에게 객체를 만들라고 말하는 언어가 아니다. 객체, 클래스, 타입, 런타임 사이의 약속을 명시하라고 요구하는 언어에 가깝다.

우리가 배워야 하는 것은 문법 이름의 목록이 아니라, 코드가 실행되기 전과 후에 어떤 상태가 바뀌는지 추적하는 습관이다. Java의 많은 혼란은 그 습관이 생기기 전까지는 예외처럼 보인다. 하지만 한 번 상태의 흐름으로 보기 시작하면, 헷갈리던 규칙들은 조금씩 같은 방향을 가리킨다.

겉으로 보이는 것은 메서드 호출 한 줄이다.
그 아래에서는 타입, 객체, 후보, 책임, 상태가 조용히 자리를 바꾸고 있다.
