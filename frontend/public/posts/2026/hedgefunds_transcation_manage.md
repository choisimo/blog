어느 날, 갑자기 샤워하면서 떠오른 궁금증이다 : "헤지펀드들의 실시간 수십, 수백 배 이상의 자본을 나노초 단위로 처리하면서 원자성을 어떻게 잃지 않는가?". 그래서 찾아보았다. 

우리는 대게 개념적인 학습의 관점에서, "속도를 높이면 원자성은 포기할 수밖에 없다."는 문장을 먼저 배운다.  
하지만 실제 초고속 결제 엔진은 나노초 단위 지연을 목표로 하면서도 원자성(Atomicity)을 유지한다. 어떻게 가능한 것이지? 현실에서는 트레이드 오프가 존재할 수밖에 없지 않은가?

### 원자성에 대한 근본적 이해
실무에서는 평균 지연(latency)보다 지연 변이(jitter)가 더 치명적이다.  
1ms 안에서도 실행 시점이 흔들리면 체결 가격이 어긋나고, 그 오차가 곧 손실로 전이된다.

![지연과 지터 비교](/images/2026/hedgefunds_atomicity/latency_jitter.png)
_평균 지연이 같아도 지터가 커지면 실제 체결 시점 오차가 커진다._

핵심은 원자성을 버리는 것이 아니라, **원자성을 강제하는 지점을 바꾸는 것**이다.  
락으로 실행 경로 전체를 멈추는 대신, 커밋 경계에서만 엄격한 검증을 수행한다.

## 현대 하드웨어의 한계와 실시간 locking의 역설
현대 CPU는 병렬 실행 능력이 매우 높지만, 메모리 계층과 동기화 비용은 여전히 느리다. 이 간극이 바로 메모리 월(memory wall)이다.

평균 메모리 접근 시간은 대략 다음처럼 표현된다.

`Ts = H*T1 + (1-H)*(T1+T2)`  
`H`: 캐시 적중률, `T1`: 캐시 접근 시간, `T2`: 메인 메모리 접근 시간

문제는 실시간 락 경합이 `H`를 빠르게 떨어뜨린다는 점이다.  
락 대기와 소유권 변경이 반복되면 참조 지역성(locality)이 깨지고, CPU는 연산보다 대기 시간을 더 많이 소비한다.

즉, 병렬성을 높이기 위해 넣은 락이 오히려 시스템을 직렬화한다.  
이 글에서는 그 병목이 CPU, 런타임, 데이터 모델, 분산 레이어에서 어떻게 누적되는지 순서대로 본다.

## 심층 분석: 락이 하드웨어를 어떻게 망가뜨리는가

### 2.1 지역성 파괴와 Cache Line Bouncing

CPU는 데이터를 64바이트 캐시 라인 단위로 다룬다. 여러 코어가 같은 락 변수에 쓰기를 시도하면 캐시 라인이 코어 사이를 계속 오가며 무효화된다.  
이 현상이 Cache Line Bouncing이다.

![락 경합과 캐시 라인 이동](/images/2026/hedgefunds_atomicity/cache_lock.png)
_락 변수 경합이 캐시 라인 소유권 이동을 유발해 코어 간 트래픽을 증가시킨다._

결과적으로 코어는 유효한 계산보다 캐시 일관성 트래픽 처리에 시간을 소모한다. 락 경합이 잦을수록 캐시 히트율 하락은 구조적인 문제가 된다.

![캐시 미스 증가 구조](/images/2026/hedgefunds_atomicity/cache_miss.png)
_락 경합이 반복될수록 캐시 미스와 대기 시간이 누적된다._

### 2.2 Memory Barrier와 파이프라인 정체

락은 메모리 순서 보장(memory ordering)을 강제하므로, CPU의 추측 실행과 out-of-order 최적화를 자주 멈추게 한다.

![메모리 배리어와 파이프라인 정지](/images/2026/hedgefunds_atomicity/memory_barrier.png)
_메모리 배리어는 실행 순서를 강제하는 대신 파이프라인 병렬성을 낮춘다._

![메모리 배리어 상세 흐름](/images/2026/hedgefunds_atomicity/memory_barrier2.png)
_배리어 전후 명령 실행 흐름의 차이와 직렬화 구간을 비교한 도식._

스토어 버퍼를 비우는 순간 파이프라인 정체(stall)가 발생하고 IPC(Instructions Per Cycle)가 급락한다.  
락은 단순 대기 메커니즘이 아니라, CPU가 쌓아둔 실행 최적화를 반복적으로 초기화하는 장치가 된다.

![IPC 하락과 배리어 영향](/images/2026/hedgefunds_atomicity/ipc_mem_barrier.png)
_배리어 개입이 잦아질수록 IPC가 하락해 유효 처리량이 감소한다._

![배리어 구간별 IPC 변화](/images/2026/hedgefunds_atomicity/ipc_memory_barrier.png)
_메모리 순서 보장 구간이 늘어날수록 IPC가 단계적으로 떨어지는 패턴._

### 2.3 RCU: 읽기 경로에서 동기화 비용을 제거하는 전략

읽기 비중이 높은 시스템에서는 단순 lock-free보다 RCU(Read-Copy-Update)가 더 실용적일 때가 많다.

- 읽기 경로: 락 없이 진행
- 쓰기 경로: 복사 후 새 버전으로 원자적 교체
- 해제 경로: grace period 이후 백그라운드에서 정리

대신 구버전이 잠시 메모리에 남아 메모리 사용량이 늘어난다.  
이 트레이드오프는 p99 지연 안정성이 중요한 결제/리스크 조회 경로에서 충분히 가치가 있다.

## 런타임 아키텍처: Shared-Nothing과 Thread-per-Core

락 최적화만으로는 한계가 있다.  
근본 원인은 "많은 스레드가 같은 상태를 공유"하는 구조 자체이기 때문이다.

### 3.1 컨텍스트 스위칭: 고성능 시스템의 Programmed I/O 퇴행

락으로 스레드가 블록되면 OS는 컨텍스트 스위칭을 수행한다.  
레지스터 저장/복원, 커널 모드 전환, 스케줄링 결정이 반복되며 지연이 누적된다.

![컨텍스트 스위칭 오버헤드](/images/2026/hedgefunds_atomicity/context-switching.png)
_문맥 전환은 순수 연산이 아닌 상태 저장/복원 비용을 강제한다._

![TLB flush와 주소 변환 재구성](/images/2026/hedgefunds_atomicity/flush_tlb.png)
_스케줄링 전환 시 TLB 재구성이 추가 지연을 만든다._

초저지연 엔진 기준에서 이 비용은 결코 작지 않다.  
더 큰 문제는 CPU가 "업무 계산" 대신 "스케줄링 관리"에 시간을 쓰게 된다는 점이다.

### 3.2 Thread-per-Core와 SPSC 링버퍼

Shared-Nothing + Thread-per-Core 모델은 코어 간 공유 상태 자체를 줄인다.

- 각 코어는 전용 스레드 하나만 실행
- 코어 간 통신은 SPSC(Single-Producer Single-Consumer) 링버퍼로 메시지 전달
- 캐시 라인 정렬로 헤드/테일 포인터 간 간섭 최소화

이 구조의 효과는 명확하다.

- 커널 개입 감소로 컨텍스트 스위칭 비용 완화
- 코어별 캐시 지역성 보호
- 계좌/자산 샤딩 전략이 아키텍처의 중심으로 이동

![SPSC 큐 기반 코어 간 통신](/images/2026/hedgefunds_atomicity/spsc_queue.png)
_공유 락 대신 단방향 큐 메시징으로 코어 간 간섭을 줄인다._

![데이터 샤딩 매트릭스](/images/2026/hedgefunds_atomicity/data_sharding_matrix.png)
_계좌/자산 단위 샤딩 설계가 처리량과 충돌률을 결정한다._

## 소프트웨어 공학 관점: 제어 경로와 데이터 경로의 분리

런타임을 바꿨다면 데이터 갱신 방식도 바뀌어야 한다.

### 4.1 DMA 효율성과 동기적 락킹의 충돌

DMA는 CPU 개입 없이 I/O와 메모리 전송을 수행해 병렬성을 높인다.  
하지만 소프트웨어가 동기적 락 대기로 흐르면 CPU는 결국 블록되고, DMA 이점은 상쇄된다.

하드웨어가 비동기적으로 빨라져도 소프트웨어가 동기적으로 막히면 전체 시스템은 느려진다.

### 4.2 Append-only 모델과 이벤트 소싱

In-place update는 동일 주소에 대한 경쟁 쓰기를 만든다.  
이를 피하는 대표 전략이 Append-only + 이벤트 소싱(Event Sourcing)이다.

- 상태를 직접 덮어쓰지 않고 이벤트를 추가 기록
- 기존 레코드는 불변(immutable) 유지
- 재처리 안전성을 위해 시퀀스 + 멱등성(idempotency) 설계 적용

![이벤트 소싱 처리 흐름](/images/2026/hedgefunds_atomicity/event_sourcing.png)
_명령, 이벤트 기록, 상태 재구성이 분리된 append-only 처리 모델._

![In-place 충돌 vs Materialized View](/images/2026/hedgefunds_atomicity/in-place-collision_materialized.png)
_덮어쓰기 충돌 모델과 이벤트 소싱 기반 조회 모델의 차이._

이 방식은 쓰기 경합을 줄이는 대신, 조회 경로의 재구성 비용(event lag)을 관리해야 한다.  
실무에서는 materialized view, 스냅샷, 체크포인트로 이를 보완한다.

## 글로벌 분산 시스템: 강한 정합성의 포기와 수렴 모델

단일 노드에서 해결된 병목은 네트워크 경계에서 다시 나타난다.

### 5.1 분산 락의 기회 비용

캐시/메모리 지연(ns)과 네트워크 왕복 지연(µs~ms) 사이에는 수천~수만 배 차이가 있다.  
글로벌 환경에서 강한 정합성을 분산 락으로 강제하면 합의 과정의 직렬화 비용이 지배적이 된다.

결과적으로 처리량은 가장 느린 노드와 네트워크 상태에 종속된다.  
HFT 결제 엔진에는 치명적인 구조다.

### 5.2 CRDT: 수학적 수렴으로 정합성을 달성하는 방법

CRDT(Conflict-free Replicated Data Type)는 충돌을 막는 대신, 충돌 후에도 상태가 수렴하도록 데이터 타입을 설계한다.

- 교환법칙(Commutativity)
- 결합법칙(Associativity)
- 멱등성(Idempotency)

이 조건이 성립하면 도착 순서가 달라도 최종 상태는 동일해진다.

다만 비가환 연산(예: 선착순 체결 우선순위, 레버리지 한도 즉시 차단)은 CRDT만으로 표현하기 어렵다.  
그래서 실무는 "강한 정합성이 필요한 핵심 구간만 좁게 직렬화"하고, 나머지는 수렴 모델로 처리한다.

## 종합: 레이어별 병목 분해와 아키텍처 트레이드오프

레이어별 병목을 압축하면 아래와 같다.

| 레이어 | 주요 병목 | 대표 지연 범위 | 핵심 오버헤드 |
|---|---|---|---|
| CPU/Cache | Cache line bouncing | ns | 캐시 일관성 트래픽 |
| Memory | Memory barrier | 수십~수백 ns | 스토어 버퍼 플러시, 파이프라인 정체 |
| OS | Context switch | µs | 커널 전환, 스케줄링 |
| Network | Distributed coordination | µs~ms | 합의/복제 직렬화 |

대안별 트레이드오프도 분명하다.

| 대안 | 성능 | 정합성 모델 | 운영 복잡도 |
|---|---|---|---|
| Shared-Nothing (Thread-per-Core) | 매우 높음 | 코어 내부 강한 정합성 | 샤딩/재배치 설계 필요 |
| Append-only + Event Sourcing | 높음 | 최종 정합성 중심 | 뷰 동기화/재생 비용 관리 필요 |
| CRDT | 분산 환경에서 우수 | 수학적 수렴 | 비가환 비즈니스 규칙 처리 난도 높음 |

요점은 단순하다.  
성능이 올라가면 정합성이 사라지는 것이 아니라, **정합성을 구현하는 방식이 직렬화에서 검증/수렴으로 이동**한다.

## 1. 결제 원자성은 '영구 저장소'를 묶는 것이 아니라, '원장 레코드'를 묶는다

초레버리지 결제 한 건은 보통 다음 상태를 함께 바꾼다.

- 가용 잔고
- 증거금 잠금 상태
- 포지션 수량/평균단가
- 수수료 및 펀딩/이자 계정
- 주문 상태(NEW → MATCHED → SETTLED)

이 중 일부만 반영되면 시스템은 즉시 불일치 상태가 된다.  
그래서 실전 엔진은 이를 개별 커밋하지 않고 `debit+margin+position+event`를 **하나의 원장 단위**로 확정한다.

즉, 속도 때문에 원자성을 버리는 것이 아니라, 속도를 위해 원자성 경계를 더 명확히 정의한다.

## 2. Lock-free가 되는 이유: 락이 없어도 충돌은 CAS에서 검출할 수 있기 때문이다

전역 락 대신 버전 기반 CAS(compare-and-swap) 검증을 사용하면 충돌을 대기 없이 감지할 수 있다.

1. 스냅샷(값 + version) 읽기
2. 변경 델타 계산
3. CAS로 version 교체 시도
4. 실패 시 최신 스냅샷으로 재계산 후 재시도
5. 성공 시 WAL/체크포인트 반영

![CAS 기반 충돌 감지와 커밋](/images/2026/hedgefunds_atomicity/cas.png)
_락 없이도 CAS 실패/재시도로 충돌을 감지하고 원자 커밋을 보장한다._

핵심은 실패를 "오류"가 아니라 "충돌 감지 신호"로 다루는 것이다.  
이 방식은 WAIT 구간을 줄여 고트래픽 구간에서도 처리량을 유지한다.

## 3. 왜 ‘원자적 커밋’이 가능한가: 두 단계가 아닌 ‘검증 기반 단일 커밋’

MVCC와 버전 모델의 핵심은 중간 상태 노출이 아니라, **커밋 시점의 일괄 검증**이다.

- 읽기 시점 버전 보관
- 스냅샷 기반 계산
- 커밋 직전 규칙 검증(버전, 마진 규칙, 우선순위)
- 통과 시 한 번에 확정, 실패 시 전체 재시도

따라서 외부에서 관측 가능한 상태는 "성공한 커밋"뿐이다.  
이것이 lock-free 구조에서도 원자성을 유지할 수 있는 이유다.

## 4. 결제의 “완전한 원자성”은 어디에서 깨지는가

엔진 내부(인메모리 장부, 내부 정산 큐, 복제 로그 순서)에서는 강한 원자성을 유지할 수 있다.  
깨지는 지점은 외부 연동 경계다.

예를 들어 은행, CCP, 청산사, 외부 지수 공급자 API가 끼면 물리적으로 단일 원자 커밋을 만들기 어렵다.  
그래서 실무는 보통 다음으로 분리한다.

`내부 원장 원자성 + 멱등 이벤트 + 보상(Compensation) 정합성`

이 방식은 고전적 2PC보다 운영 복원력이 높고 지연도 낮다.

## 5. 왜 이게 결제에서 특히 중요해지는가 (초레버리지 특성)

초레버리지 환경은 두 민감도가 동시에 높다.

- 시간 민감도: 지연이 곧 슬리피지와 기회비용
- 정합성 민감도: 작은 불일치가 연쇄 청산 리스크로 증폭

CAS 기반 lock-free + MVCC + 선형 커밋은 이 두 조건을 함께 맞춘다.

- 지연 최소화: 락 경합 제거
- 정합성 보존: 실패-재시도로 충돌 해소
- 추적 가능성: tx id + 순차 로그로 재연 가능

## 6. 정리

나노초 단위 결제에서 원자성이 유지되는 이유는 시간을 "마법처럼 줄여서"가 아니다.  
**대기 비용을 줄이고, 커밋 경계를 엄격히 통제하도록 책임을 재배치했기 때문**이다.

1. 핵심 상태를 하나의 원장 단위로 원자 갱신한다.
2. 락 대신 CAS/버전 검증으로 충돌을 감지하고 재시도한다.
3. 커밋 시점의 단일 검증으로 중간 상태 노출을 차단한다.
4. 외부 연동은 멱등 이벤트와 보상 절차로 정합성을 회복한다.

결론적으로 "속도 vs 무결성"은 고정된 대립축이 아니다.  
초고속 결제 엔진은 동시성은 느슨하게 열고, 커밋 경계는 강하게 닫는 방식으로 두 목표를 동시에 달성한다.

## 보충: 불변 인프라가 원자성의 운영 조건이 되는 이유

지금까지는 트랜잭션 엔진 내부의 동시성 제어를 다뤘다.  
하지만 운영 인프라가 가변적(mutable)이라면 엔진이 아무리 정교해도 지터와 재현 불가 장애를 피하기 어렵다.

### A. Mutable 인프라가 만드는 비결정론

운영 중 서버를 계속 수정하는 방식은 노드 간 설정 드리프트(configuration drift)를 만든다.  
그 결과 동일 주문 흐름도 노드마다 다른 경로를 타며 지연 편차가 커진다.

![패치 누적과 설정 드리프트](/images/2026/hedgefunds_atomicity/drift_patching.png)
_가변 운영은 시간이 지날수록 노드별 상태 편차를 키운다._

| 드리프트 원인 | HFT 리스크 | 완화 전략 |
|---|---|---|
| 수동 패치, 실시간 라이브러리 교체 | 지터 증가, 재현 불가 버그 | 이미지 단위 재배포, 동시 교체 |
| 운영 중 커널/네트워크 파라미터 수정 | 노드별 경쟁 상태 차이 | IaC 기반 선언적 설정 강제 |
| 개발/운영 환경의 미세한 커널 차이 | 예측 불가 halt, I/O 병목 | 불변 컨테이너 이미지로 환경 일치 |

### B. 저수준 관점: 인터럽트, 문맥 전환, DMA

초저지연 구간에서는 인터럽트 한 번도 부담이 된다.  
인터럽트가 발생하면 PSW/PC 저장과 컨텍스트 전환이 뒤따르고, 이 오버헤드는 지터로 관측된다.

그래서 일부 경로는 인터럽트 기반 처리 대신 폴링이나 전용 코어 전략을 선택한다.  
또한 대량 데이터 이동은 DMA로 넘겨 CPU가 거래 로직 연산에 집중하도록 분리한다.

### C. 왜 '수정'이 아니라 '교체'인가

컨테이너는 namespace/cgroups를 통해 실행 환경을 격리하고 자원 경계를 강제한다.  
핵심은 배포 방식이다. 운영 중 컨테이너 내부를 수정하지 않고, 검증된 이미지를 새 버전으로 통째로 교체해야 한다.

![불변 배포 상태 모델](/images/2026/hedgefunds_atomicity/immutable_state.png)
_수정이 아닌 교체 중심 배포가 상태 일관성과 롤백 가능성을 높인다._

이 모델의 장점은 세 가지다.

- 재현성: 개발/스테이징/운영의 실행 경로를 최대한 동일화
- 복구성: 장애 시 이전 이미지로 즉시 롤백
- 보안성: 런타임 변경 지점을 줄여 침투 표면 축소

### D. 쿠버네티스: 클러스터 단위의 선언적 일관성

노드 수가 늘어나면 단일 서버 불변성만으로는 부족하다.  
쿠버네티스는 desired state를 기준으로 드리프트를 감시하고, 장애 노드를 "수리"하지 않고 "교체"한다.

![Desired State와 Self-healing 루프](/images/2026/hedgefunds_atomicity/desired_state.png)
_선언된 목표 상태와 실제 상태의 차이를 자동으로 보정한다._

즉, 트랜잭션 엔진의 원자성 경계를 데이터 레이어에서 지킨다면, 쿠버네티스는 운영 레이어에서 상태 경계를 지킨다.  
둘은 별개 기술이 아니라 같은 목표(결정론적 예측 가능성)를 위한 상호보완 구조다.

물론 대가도 있다. 컨트롤 플레인 복잡도 증가, 커널 공유 리스크, 디버깅 난도 상승이 따른다.  
그럼에도 HFT 환경에서는 인적 수정으로 인한 비결정론 리스크가 더 크기 때문에, 불변-교체 모델이 전략적으로 유리하다.
