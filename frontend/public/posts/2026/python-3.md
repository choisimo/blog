1,2부를 통해 파이썬의 근간인 `PyObject` 구조부터 시작해 딕셔너리 기반의 네임스페이스(`__dict__`), 전용 메모리 할당기(`pymalloc`), 그리고 객체의 생명주기를 관리하는 참조 카운트와 가비지 컬렉터(GC)까지 살펴보았습니다.

결국, 파이썬의 핵심인 "모든 것을 동적 객체로 다루는" 개념은 개발자에게 엄청난 생산성과 유연성을 주지만, 동시에 C 언어 레벨로 내려가면서 끊임없는 포인터 추적, 메모리 할당 및 해제 오버헤드, 그리고 떨어지는 캐시 지역성(Cache Locality)이라는 막대한 비용을 치르고 있다는 점을 알게되었고,

이런 무거움을 줄이기 위해 CPython에서는 `pymalloc`, 문자열 인터닝, 적응형 바이트코드 같은 꼼수들을 사용한다는 것을 확인했습니다. 하지만, 이런 꼼수들만 믿고 코드를 함부로 짜면 인터프리터의 최적화 노력이 의미가 없어질 겁니다. 그렇다면 우리는 어떻게 코딩해야 이 무거운 동적 객체의 늪을 피하고 엔진의 성능을 끌어낼 수 있을까요?

따라서, 이번 편에서는 **절대 피해야 할 안티 패턴(Anti-pattern)**과 **올바른 최적화 패턴**, 그리고 파이썬 병렬 처리의 영원한 화두인 **GIL(Global Interpreter Lock)**에 대해 살펴보겠습니다.

---


## 1. 객체 Churn의 늪: 루프 안에서의 무의미한 생성과 파괴

CPython에서 가장 비싼 작업 중 하나는 바로 **객체를 새로 만들고 부수는 일(Object Churn)**입니다. 앞서 살펴 보았듯이 메모리 할당(`pymalloc`), 참조 카운트 증감, GC 추적 등 보이지 않는 수많은 C 레벨 작업이 동반되기 때문입니다.

### 문자열 `+=` 연산
가장 대표적인 실수가 루프 안에서 문자열을 `+=`로 이어붙이는 것입니다. 파이썬에서 문자열(`str`)은 불변(Immutable) 객체입니다. 따라서 `s += p`를 실행할 때마다 기존 문자열을 버리고, 두 문자열의 크기를 합친 **새로운 문자열 객체를 힙 메모리에 계속 새로 할당** 하게되는 치명적인 연산 작용이 발생합니다. 

![](/images/2026/python-3/2.png)
```python
# 잘못된 예시: 매 루프마다 새로운 객체가 생성되고 버려짐
def bad_concat(parts):
    s = ""
    for p in parts:
        s += p  # 객체 Churn 발생 지점
    return s
```

### 대안 : `join()`을 통한 C 레벨 일괄 연산(Bulk Operation)

```python
# 올바른 예시: 필요한 메모리를 한 번에 계산하여 일괄 할당
def good_concat(parts):
    return "".join(parts)
```

`"".join()`이 구조적으로 빠른 이유는 명확합니다. 표준 라이브러리의 C 구현체가 리스트를 한 번 쭉 훑어서 **전체 길이를 미리 계산한 뒤, 단 한 번의 메모리 할당만으로 문자열을 조립**하기 때문입니다. 중간 단계의 쓰레기 객체가 전혀 생성되지 않습니다.

## 2. 디스패치 오버헤드: 루프와 함수 호출의 덫

1부에서 확인했듯, 파이썬의 메서드 호출은 객체의 `__dict__`를 뒤져 함수 포인터를 찾고(`LOAD_ATTR`), 스택에 인자를 쌓은 뒤 실행하는(`CALL_METHOD`) 무거운 과정(Dispatch)을 거칩니다.

### 루프 안에서 `.append()` 남발

```python
def bad_loop(nums):
    out = []
    for x in nums:
        out.append(x * 2)  # 매번 append 속성을 찾고 함수를 호출함
    return out
```

이 코드가 느린 이유는 명확합니다. 루프를 돌 때마다 리스트 객체의 네임스페이스를 뒤져 `append`라는 메서드를 찾고, 파이썬 레벨의 함수 호출 오버헤드를 계속 발생시키기 때문입니다.

![](/images/2026/python-3/1.png)

### 대안 : 리스트 컴프리헨션 (C-level Fast Path)

```python
def good_loop(nums):
    return [x * 2 for x in nums]
```

리스트 컴프리헨션은 단순한 문법적 설탕(Syntactic Sugar)이 아닙니다. 바이트코드로 뜯어보면 `bad_loop`에 있던 `LOAD_ATTR`와 `CALL` 계열 호출 오버헤드가 줄고, `LIST_APPEND` 중심의 전용 경로가 쓰이는 것을 확인할 수 있습니다. CPython 버전에 따라 주변 opcode는 달라질 수 있지만, 파이썬 레벨의 메서드 호출을 우회하고 CPython 내부의 고속 경로(Fast Path)를 직접 타기 때문에 압도적으로 빠른겁니다.

## 3. GIL (Global Interpreter Lock): 스레드라는 환상

파이썬 아키텍처를 이야기할 때 GIL(Global Interpreter Lock)을 빼놓을 수 없습니다. 멀티 코어 시대에 파이썬은 왜 여전히 하나의 락(Lock)에 의존하고 있는 것일까요?

2부에서 배운 `PyObject`의 '참조 카운트(Refcount)'를 떠올려 봅시다. 만약 여러 OS 스레드가 동시에 하나의 파이썬 리스트에 접근해서 요소를 추가하고 삭제한다면, 참조 카운트에 경쟁 상태(Race Condition)가 발생하여 메모리가 엉망이 될 것입니다.

![](/images/2026/python-3/3.png)
CPython은 이 문제를 아주 무식하지만 확실한 방법으로 해결했습니다. **"한 번에 오직 하나의 스레드만 파이썬 바이트코드를 실행할 수 있도록 거대한 자물쇠(GIL)를 채우자."**



### CPU-bound 작업에 스레드 풀 사용

```python
from concurrent.futures import ThreadPoolExecutor

# 잘못된 선택: 순수 연산(CPU-bound) 작업에 스레드를 쓰면 오히려 느려짐
with ThreadPoolExecutor() as ex:
    list(ex.map(heavy_math_calculation, data))
```

위 코드를 실행하면 OS는 스레드들을 여러 코어에 분배하지만, 정작 스레드들은 GIL을 얻기 위해 서로 병목(Contention)을 일으킵니다. 코어는 여러 개가 돌고 있는데 실제 연산 속도는 싱글 코어와 같거나 락 스위칭 비용 때문에 오히려 더 느려집니다.

### 대안 : 작업의 성격에 맞는 병렬성 도구 선택

  * **I/O Bound 작업 (네트워크 통신, 파일 읽기/쓰기):** `ThreadPoolExecutor` 또는 `asyncio`를 사용합니다. 파이썬 스레드는 외부 응답을 기다리는 동안(Blocking I/O) 자발적으로 GIL을 내려놓기 때문에 스레딩의 이점을 충분히 누릴 수 있습니다.
  * **CPU Bound 작업 (수학 연산, 이미지 처리):** `ProcessPoolExecutor`를 사용하여 OS 레벨의 프로세스를 여러 개 띄워야 합니다. 각 프로세스는 자신만의 독립된 CPython 인터프리터 메모리 공간과 자신만의 GIL을 가지므로 완벽한 멀티 코어 병렬 처리가 가능합니다. (또는 파이썬 3.14+ 부터 본격화되는 멀티 인터프리터 기능을 기대해 볼 수 있습니다.)

-----

결국 GIL은 파이썬이 동시성의 비용을 숨기기 위해 선택한 안전장치에 가까운 것입니다. 그래서 실제 사용 시에는 "스레드로 더 빨라질 수 있는가"라는 뗌질이 아니라, "작업이 I/O-bound인지 CPU-bound인지"를 먼저 나누고, 시작하는 것이 장기적으로 가장 효율적이라 봅니다.
