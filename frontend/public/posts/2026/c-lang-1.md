## 1. 왜 포인터 같은 이상한 것을 배우는가?

아마 많은 사람들이 코딩을 배우기 위해서 야심차게 기본이 최고지 하면서 
C언어로 입문했을 때가 있었을 것이다. 대부분의 경우 for, while 까지는 그럭저럭 
안다고 착각하고, 포인터를 들어갈 때 분명 간단한 기호들만 사용하는데 아무리 봐도 
건빵 속 별사탕으로만 보여서 포기했던 사람들도 많을 것이다. 나도 그랬었다. 

### 배열의 선언과 사용 
C에서 배열을 한 문장으로 요약하면, **"같은 타입의 원소 N개를 연속된 메모리에 고정 배치한 객체"** 라고 할 수 있다. 여기서 주목해야할 점이 같은 타입의 원소, 고정 배치라는 점이다. 

#### 배열의 선언
```c
int arr[5]; 
```
이러한 **int 타입**의 배열을 선언하는 순간, 메모리에는 sizeof(int) * 5 크기의 연속된 공간이 고정적으로 잡힌다. 앞서 주목했던 '**같은 타입의 원소**', '**고정 배치**' 가 적용되기 때문에
arr 는 단순한 변수명이 아니라, 이 연속된 메모리 블록 전체를 식별하는 고유한 이름이자 
**객체** 인 것이다. 

![](/images/2026/c-array-1/1.png)

흔히 입문 때 배열을 "값 여러 개를 담는 상자" 정도로 설명한다. 물론 입문 단계에서는 나쁘지 않은 비유겠지만, 나노초 단위의 연산을 수행해야하는 컴파일러와 CPU 관점에서는 말도 안 되는 소리로 들릴 것이다. 
![](/images/2026/c-array-1/2.png)
그들에게 배열이란 단순한 문법 설탕이 아니라, **원소 타입과 길이가 타입 시스템에 녹아 있는 실제 객체** 이다. 기초 골격부터 안다라는 편안한 착시에 빠져 있었던 것이다.  

#### 배열의 사용

![](/images/2026/c-array-1/3.png)
이 그림을 보며 우린 배열의 '몇 번째 칸을 꺼낸다' 라고 생각한다.
하지만 그런 추상적인 마법은 존재하지 않는다.
```c
// *(arr + 2) = 10; 과 동일하다.
arr[2] = 10; 
```
이처럼 수식 안에서 arr 라는 이름을 값을 읽거나 쓰기 위해서 사용할 때, C 컴파일러는 이를 '배열의 첫 번째 원소를 가리키는 주소값'으로의 Array Decay 를 일으킨다. 

![](/images/2026/c-array-1/4.png)
즉, arr[i] 라는 문법은 그냥 *(arr + i) 를 보기 좋게 만든 껍데기일 뿐이라는 것이다. (arr 는 &arr[0]) 

이것이 C 배열이 외부와 소통하는 인터페이스 방식이다. 무거운 배열 전체를 복사해서 넘기는 대신, 배열의 첫 시작점(Base Address)만 알려주고 나머지 원소들은 포인터 연산(Pointer Arithmetic)을 통해 접근하도록 한다.  

배열의 시작 주소를 넘긴 뒤, 컴파일러가 Base + offset 형태의 주소 계산을 생성하고, CPU 가 그 주소에 대해 Load/Store 명령을 수행한다. 여기서 더 확장해서 생각해보면 동적 할당 문법도 유추해낼 수 있다.
```c
int *p = malloc(sizeof(int) * 5); 
```
int *p 는 int 원소를 가리킬 포인터 변수 p 
malloc(sizeof(int) * 5) 는 int 5개가 들어갈 만큼의 연속된 바이트 공간을 힙에 확보
반환값은 그 메모리 블록의 Base Address.

![](/images/2026/c-array-1/5.png)
결국에 arr 는 배열 그 자체이고, p 는 그 배열 비슷한 메모리 블록의 Base Address 를 들고있는 포인터이다. 비유하자면, arr 는 기차 전체 편성 자체이고, arr 가 수식에서 쓰일 때는 첫 번째 칸의 위치 정보, p 는 실체가 아니라 주소를 저장한 별도 변수이기에 첫 번째 칸 위치를 적어둔 메모지라고 기억하면 된다. 왜냐하면 p 자체는 메모리 한 칸짜리 포인터 변수이고, 그 포인터가 가리키는 대상은 따로 존재하기 때문이다.

#### 과거 회상
저학년 시절에 교수님과 상담을 했던 적이 있다. 내가 이제야 좀 포인터를 알겠다고 했더니 "그럼 배열은 포인터인 것 알겠네?" 라고 말씀하셔서 내가 "그렇습니다." 라고 답변했었다. 이는 함정이었던 것 같다. 
맞는 대답은 다음과 같았을 것이다. "배열의 정체성은 포인터가 아닙니다. 단지 대부분의 표현식 문맥에서 첫 번째 원소를 가리키는 포인터로 변환(Decay) 되어 동작할 뿐 입니다."


### '대부분의' 표현식 문맥에서만.. 
인지 못 했다면 다시 읽어보자. 배열은 **항상** 첫 원소를 가리키는 포인터로 **자동 변환** 된다고 안 적혀있다. 그럼 언제 붕괴(Decay)하지 않는 것일까? 컴파일러는 어떠한 경우에 배열 전체 그 자체를 알아야할까?
```c
int arr[5];
```
식 안에서 arr를 사용하면, 컴파일러는 보통 이것을 
```c
&arr[0] 
```
처럼 첫 번째 원소의 주소로 바꿔서 처리한다. Decay, 풀네임으로는 array-to-pointer-decay 라고 부른다. 그럼 Decay가 필요한 이유가 뭘까?
arr + 1 을 하려면 컴파일러는 arr 를 주소처럼 해석해야 한다. 그래서 arr 는 &arr[0]로 decay 한다가 지금까지 살펴본 내용이었다. 따라서 arr + 1 은 사실상 *arr[0] + 1 이고, 결과는 arr[1] 의 주소이다. 

왜 arr[1] 이냐고 생각할 수 있다. CPU 입장에서 생각해야 이해된다. 
산술 연산은 배열 전체 객체에 하는 것이 아니라, 주소 값에 하는 것이기에 그렇다.

### 좋다, 그럼 Decay 는 왜 쓰는지 알겠는데, 붕괴하지 않을 때는 언제인가?

#### sizeof로 본질 찾기
**sizeof arr** 과 같은 크기 측정 때에는 컴파일러가 알아야하는 것은 첫 원소의 주소가 아니다.
그냥 **배열 전체의 바이트 크기**만 알면 된다. 

우린 몸무게 잰다고 발 무게, 머리 무게, 몸통 무게...이렇게 측정하지는 않습니다. 
쓸모없는 정보들은 다 쳐내고 편하게 체중계에 올라 몸무게 그 자체를 재고 싶거든요. 
```
int arr[5];
printf("%zu\n", sizeof(arr));
```
결과는 sizeof(int) * 5 입니다. 만약 decay 가 일어나서 int * 가 되면 sizeof(arr)는 포인터 크기, 즉 4바이트 혹은 8바이트 크기만 나오게 되는거죠. 

#### 주소 연산자 (&) 를 만났을 때
```c
int arr[5];
// arr 의 타입: int * (첫 번째 원소를 가리킴)
// &arr 의 타입: int (*)[5] (크기가 5인 int 배열 전체를 가리킴)
```
%p 으로 출력해보면 arr, &arr 의 시작 메모리 주소값 자체는 동일하게 나옵니다.
하지만 이는 완전히 같은게 아니라 같아 보이는 착시일 뿐이죠.
- arr + 1 : 다음 원소로 이동한다. (sizeof(int) 인 4바이트 만큼만 이동)
- &arr + 1 : 배열 전체 크기만큼 이동한다. (sizeof(int) * 5 인 20바이트 만큼 이동)

즉, 기차의 '첫 번째 객차의 시작 위치'나 '기차 전체의 시작 위치'나 물리적인 좌표는 같지만
연산을 수행하는 순간 그 스케일의 차이가 나타납니다.
만약 Decay 가 일어났다면 &(&arr[0]) 같은 형태가 되어 rvalue 주소를 
구하려다가 컴파일 에러가 발생했을 겁니다. 이를 실생활에서의 비유하자면 다음과 같을 것 입니다.
```text
A : “집 주소 알려줘”
B : “여기 현관문 좌표요”
A : “아니, 그 현관문 좌표가 적힌 종이의 주소를 달라고”
B : “혹시 다른 시공간 좌표계를 사용하시는 겁니까?”
```

#### 문자열 리터럴로 문자 배열을 초기화할 때
우리가 흔히 사용하는 문자열 초기화 방식에서도 배열은 붕괴하지 않습니다.
```c
char str[] = "Hello";
```
우변의 "Hello"는 내부적으로 char[6] 타입의 배열(문자열 리터럴)입니다.
평소 같으면 이 역시 수식 안에서 첫 번째 문자인 'H'의 주소(char *)로 붕괴해야 맞지만,
배열을 초기화하는 이 문맥에서는 포인터로 붕괴하지 않습니다.

대신 붕어빵 틀에서 붕어빵을 찍어내듯, 새로 생성되는 str 배열 공간에
'H', 'e', 'l', 'l', 'o', '\0' 이라는 원소들을 하나씩 복사해 넣는 
원본 데이터로서의 정체성을 유지합니다. 


### 차원을 잃어버린 배열의 억울함
C언어에서 배열이 가장 억울해지는 순간은 바로 함수의 인자(Parameter)로 전달될 때 입니다.
![](/images/2026/c-array-1/6.png)

분명 배열 전체를 넘겼다고 생각했는데, 함수 내부에서는 갑자기 배열이 쪼그라들어 버리는 기이한 현상이 
발생합니다.
```c
void printArraySize(int arr[5]) {
    // 여기서 결과는 20이 아니라 4 혹은 8 (포인터 크기)이 나옵니다.
    printf("함수 내부: %zu\n", sizeof(arr)); 
}

int main() {
    int myArr[5];
    printf("main 함수: %zu\n", sizeof(myArr)); // 여기서는 정상적으로 20
    printArraySize(myArr);
    return 0;
}
```
매개변수에 명시적으로 int arr[5]라고 정성껏 적었음에도 불구하고, 컴파일러는 이를 무자비하게 int *arr로 강제 변환시켜 버립니다. 함수 입장에서는 배열의 전체 크기(차원 정보)를 완전히 잃어버리고, 그저 **시작 주소만 가리키는 메모지(포인터)**만 쥐게 되는 것입니다. 

왜 이런식으로 만든거야라는 생각대신 C 언어가 생겨났을 시기의 컴퓨팅 환경을 생각해보면 이해가 갑니다.
지금은 보잘 것 없어보이는 크기의 배열도 함수에 넘길 때마다 메모리에 복사 (call by value) 한다면,
그 당시 컴퓨터는 비명횡사 하면서 뻗어버렸을 것 입니다.

그래서 C언어는 배열을 함수로 넘길 때 무조건 '시작 주소'만을 넘기는 방식을 선택했고,
C 언어나 C++ 에서 반드시 배열의 길이 (size) 를 별도의 매개변수로 함께 넘겨주어야 한다는 
당연한 절차가 생겨난 것 입니다.


### 2차원 배열의 진짜 메모리 구조
1차원 배열의 Decay 메커니즘을 이해했다면, 차원을 하나 더 올려봅시다.
우리는 보통 2차원 배열을 엑셀 스프레드시트나 바둑판의 격자 (Grid) 모양으로 상상합니다.

근데, 컴퓨터 메모리에는 다차원이라는 물리적 공간 자체가 존재하지 않습니다. 
HBM 같은 경우에는 패키징/구조의 3차원이지, 주소 공간이 x,y,z 축으로 펼쳐진다라는 의미는 아닙니다.
즉, 데이터가 다차원일 수는 있어도, 컴퓨터는 아직 주소 공간을 다차원적으로 처리 못 합니다.

메모리는 오직 1차원의 길고 평면적인 선형 공간일 뿐 입니다.
그렇다면 arr2d는 어떻게 메모리에 저장될까요? 

![](/images/2026/c-array-1/9.png)
C언어에서 2차원 배열의 진짜 정의는 **"배열을 원소로 가지는 배열(Array of Arrays)"**입니다. 
```c
int arr2d[3][4];
```
는 단지 "크기가 4인 int 배열"이 3개 연속으로 일렬로 붙어 있는 1차원 메모리 블록일 뿐입니다.

#### 2차원 배열은 어떻게 Decay 되는 것일까?
그렇다면 2차원 배열의 이름인 arr2d가 수식 안에서 쓰일 때, 어떤 타입의 포인터로 붕괴할까요? 
2차원 배열이니 자연스럽게 더블 포인터(int **)로 붕괴할 것이라 착각하기 쉽습니다.
하지만 원칙은 언제나 동일합니다. 

> "배열의 이름은 '첫 번째 원소'를 가리키는 포인터로 Decay 된다."

arr2d의 첫 번째 원소는 무엇인가요? 정수 하나가 아니라, **int 4개짜리 배열(arr2d[0])**입니다.
따라서 arr2d는 단순한 정수 포인터가 아니라, **"크기가 4인 int 배열을 통째로 가리키는 포인터"**로 붕괴합니다.

이를 코드로 표현하면 int (*)[4] (배열 포인터)가 됩니다.

2차원 배열의 인덱싱 문법인 arr2d[i][j]가 내부적으로 동작할 때,
컴파일러는 이 코드를 *(*(arr2d + i) + j) 라는 포인터 연산으로 환원합니다.

    arr2d + i : i번째 행(크기가 4인 배열 전체)의 주소로 큼직하게 점프(Row Offset).

    *(arr2d + i) : 그 행의 첫 번째 원소 주소(int *)로 한 단계 Decay 됨.

    ... + j : 그 주소에서 j번째 열만큼 자잘하게 점프(Column Offset).

    *( ... ) : 최종적으로 그 위치에 도달해 값을 읽거나 씀.

결국 다차원 배열이란, 인간이 다루기 편하도록 컴파일러가 포인터 연산의 보폭(Jump size)을 
다층적으로 조절해 주는 아주 정교한 문법적 환상입니다.

![](/images/2026/c-array-1/8.png)

배열은 언제나 일직선으로 누워있었고, 그것을 자르고 건너뛰며 2차원처럼
보이게 만든 것은 컴파일러의 치밀한 주소 계산이었습니다.
